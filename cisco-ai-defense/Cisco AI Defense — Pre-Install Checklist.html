01

Cisco SCC Account & AI Defense Licensing

4 items
Active Cisco Security Cloud Control (SCC) account with AI Defense entitlement
AI Defense SaaS control plane is accessed exclusively through Cisco SCC. Without an active account and entitlement the Hybrid Deployment menu will not appear.
criticalsccsaas
AI Defense is a SaaS-first product. The on-prem data plane (the hybrid connector on your AI POD) is controlled by the SaaS control plane via a bidirectional gRPC stream over TLS. You must be able to log into SCC and see the AI Defense product before any on-prem deployment can begin.
# Verify SCC access: # 1. Navigate to https://security.cisco.com # 2. Log in with your Cisco CCO credentials # 3. Look for "AI Defense" in the product list # 4. Navigate to AI Defense → Administration → Hybrid Deployment # 5. Confirm "Deploy a Connector" button is visible # If you don't see AI Defense, contact your Cisco account team # to activate the entitlement on your SCC tenant
AI Defense license covers your intended deployment size
AI Defense licensing covers applications and request rates. Confirm your license covers the number of AI applications and req/s for your AI POD size.
saas
The AI POD size determines maximum load: Small = 100 req/s / 20 apps, Medium = 200 req/s / 40 apps, Large = 300 req/s / 60 apps. Your AI Defense license must cover the applications you intend to protect.
# In SCC → AI Defense → Administration → Licenses # Verify: active license, expiry date, application count
AI Defense API key generated and stored securely
The Helm install command from the AI Defense UI requires an API key. Generate it in advance and store it — it will be inserted into the Helm command at install time.
criticalscc
The hybrid connector Helm install generates a unique API key that binds the on-prem data plane to your specific SCC tenant. This key is inserted into the Helm command provided in the AI Defense UI. If you lose this key you will need to generate a new connector.
# In SCC → AI Defense → Administration → API Keys and URLs # Generate a new API key # Store securely — treat like a password # You will be prompted to insert it during the Helm install step
Identify your AI Defense regional point of presence (PoP)
The AI Defense cloud gateway connects to the nearest regional PoP. Confirm outbound connectivity to the correct PoP endpoint from your AI POD network.
networkscc
# In SCC → AI Defense → Administration → Regional Points of Presence # Note the hostname/IP of your assigned PoP # Test outbound connectivity from jump server: $ curl -sk https://<pop-endpoint> -o /dev/null -w "%{http_code}\n" # Any response (even 4xx) confirms outbound 443 is open

03

ACI Fabric & Network Prerequisites

6 items
ACI fabric is operational and APIC is accessible
The AI POD network runs on Cisco ACI. The APIC must be reachable and the fabric healthy before any OpenShift or AI Defense networking is configured.
criticalaci
# Verify APIC is reachable from jump server: $ curl -sk https://<apic-ip>/api/node/mo/uni.json -o /dev/null -w "%{http_code}\n" # Check fabric health in APIC UI: # APIC → Fabric → Inventory → Fabric Membership # All leaf and spine nodes should be Active
VLANs and EPGs configured for OCP cluster traffic in ACI
OpenShift requires dedicated VLANs for API, ingress, internal cluster, and storage traffic. These must be defined in ACI with appropriate EPGs and contracts before OCP installation.
criticalaci
ACI EPGs control which traffic is permitted between network segments. OpenShift needs EPGs and contracts for at minimum: OCP nodes to API VIP, OCP nodes to Ingress VIP, OCP nodes to DNS, OCP nodes to internet (outbound 443 for SCC), and inter-node cluster traffic. Missing contracts will silently drop traffic causing hard-to-debug failures.
# Minimum VLANs/EPGs to confirm in ACI: # - OCP Node network (management + cluster traffic) # - API VIP VLAN # - Ingress/Apps VIP VLAN # - Storage VLAN (if applicable) # Contracts needed: # - OCP nodes ↔ Internet (outbound 443) # - OCP nodes ↔ DNS servers # - OCP nodes ↔ AI model endpoints # - Inter-node (OCP cluster traffic, all ports)
Missing ACI contracts are the most common silent failure during OCP install. Traffic drops at the fabric level with no error visible to OpenShift.
Outbound internet access on port 443 confirmed from OCP node network
The AI Defense hybrid connector connects to the SaaS control plane via TLS over port 443. This is the only required outbound port but it must be unrestricted.
criticalnetworkaci
The gRPC stream between the on-prem connector and the SaaS cloud gateway runs over TLS on port 443. If this is blocked at the ACI fabric, firewall, or proxy level the connector will appear to install successfully but will never show Active in SCC.
# Test from OCP jump server (which should be on same network as OCP nodes): $ curl -sk https://security.cisco.com -o /dev/null -w "HTTP: %{http_code}\n" $ curl -sk https://api.cdo.cisco.com -o /dev/null -w "HTTP: %{http_code}\n" # Both should return HTTP 200 or 30x (redirect) # If timeout or connection refused = ACI contract or firewall blocking port 443
DNS configured and resolving both internal and external names
OCP requires DNS for API VIP, Ingress VIP, and all node FQDNs. AI Defense requires DNS to resolve both SaaS endpoints and local AI model endpoints.
criticalnetwork
# DNS records required before OCP install: # api.<cluster-name>.<domain> → API VIP # *.apps.<cluster-name>.<domain> → Ingress VIP (wildcard) # <node-hostname>.<domain> → each node's IP # Verify DNS from jump server: $ nslookup api.<cluster-name>.<domain> $ nslookup test.apps.<cluster-name>.<domain> $ nslookup security.cisco.com # All must resolve before starting OCP install # Post-install — AI Defense DNS entries needed: # <subdomain.domain.com> → IP of OCP Ingress Controller # (added after connector is deployed)
AI model endpoints are reachable from OCP node network
AI Defense validates and protects AI models. These models (hosted on the AI POD or elsewhere) must be reachable from OCP nodes at HTTPS endpoints for Validation and Runtime to function.
network
# If model is hosted on the same AI POD: # Confirm it has an HTTPS endpoint accessible from OCP nodes # HTTP endpoints work but HTTPS is preferred per Cisco docs # Test from jump server: $ curl -sk https://<model-endpoint>/health -o /dev/null -w "%{http_code}\n" # Note the model's: # - Endpoint URL (needed during AI Defense setup) # - Request header structure # - Request payload structure (for Validation configuration)
DHCP or static IPs allocated for all OCP nodes, API VIP, and Ingress VIP
OCP requires stable IPs for all nodes plus two VIPs (API and Ingress). These must be reserved in the ACI VLAN before installation begins.
criticalnetwork
# IP allocation needed: # 3x control plane nodes (static or DHCP reservation) # 2-3x worker nodes (static or DHCP reservation) # 1x API VIP (must not be a node IP) # 1x Ingress VIP (must not be a node IP) # 1x bootstrap node (temporary, released after install) # 1x jump server # Total: 9-11 IPs minimum on the OCP network VLAN
04

Red Hat OpenShift Prerequisites

6 items
OpenShift 4.18 or higher — this is a hard minimum for AI Defense
The AI Defense hybrid connector Helm charts require OCP 4.18+. Earlier versions are not validated or supported.
criticalopenshift
This is a higher minimum than many other OpenShift workloads. OCP 4.14-4.17 will not work — the AI Defense connector requires 4.18+.
# After OCP install, verify version: $ oc version # Server Version must be 4.18.x or higher $ oc get clusterversion # Verify no update is available that would change minor version mid-install
Minimum 3 control plane + 2 worker nodes, all in Ready state
AI Defense requires a minimum of 2 worker nodes. The Helm chart distributes AI Defense components across workers. A single worker node deployment is not validated.
criticalopenshift
$ oc get nodes # Must show 3 control-plane + 2 or more workers, all Ready $ oc get co | grep -v "True.*False.*False" # No output = all operators healthy
Red Hat pull secret configured and cluster can pull images
OCP image pulls and operator installs require a valid Red Hat pull secret. Verify the cluster can pull images from registry.redhat.io before starting operator installs.
criticalopenshift
$ oc get secret pull-secret -n openshift-config -o yaml | grep -c "registry.redhat.io" # Must return 1 or more # Test by checking marketplace pods are pulling successfully: $ oc get pods -n openshift-marketplace # All should be Running
Storage class available with sufficient capacity for AI Defense workloads
AI Defense connector components may require persistent storage. A default storage class must be configured and working before Helm install.
openshift
$ oc get storageclass # At least one storageclass marked (default) must exist $ oc get pvc -A # Verify no PVCs are in Pending state (indicates storage issues)
OCP Security Context Constraints (SCCs) must be configured before Helm install
The AI Defense connector requires specific SCCs to be configured on the OCP cluster before the Helm charts are applied. The CVD documents this as a mandatory pre-step in the install procedure.
criticalopenshift
The CVD explicitly states "Before installing the helm charts for the AI Defense on AI POD deployment, these OCP Security Context Constraints must be configured." The exact SCC commands are provided in Figure 4 of the CVD via the AI Defense UI during the hybrid connector setup — they are generated dynamically for your tenant. Do not skip this step.
The specific SCC commands are generated by the AI Defense UI during the Deploy a Connector workflow. You must navigate to AI Defense → Administration → Hybrid Deployment → Deploy a Connector to get the exact commands for your tenant before running the Helm install.
# General pattern (exact commands come from AI Defense UI): # The SCC commands will look similar to: $ oc create serviceaccount ai-defense -n ai-defense $ oc adm policy add-scc-to-user <scc-name> -z ai-defense -n ai-defense # Run whatever is provided in the AI Defense UI — do not skip # Run BEFORE the helm install command
After install: also run oc adm policy add-scc-to-user anyuid -z default -n ai-defense-onprem — the Triton GPU model pods use the default service account and require anyuid to create model symlinks. Without this, modernbert and pi-mmbert Triton pods will CrashLoopBackOff. See T9 in the install guide.
Cilium CNI is healthy on all nodes including GPU worker nodes
AI Defense pods depend on Cilium for pod networking and policy enforcement. Confirm Cilium is Running on all nodes — especially any GPU workers recently added — before installing AI Defense.
criticalopenshiftnetwork
If GPU worker nodes were recently added to the cluster, the Cilium DaemonSet may not yet have scheduled on them. AI Defense pods that land on a worker without a Running Cilium agent will have broken network policy enforcement. DESIRED must equal READY.
# Check all Cilium pods are Running across all nodes: $ oc get pods -n cilium -o wide # Verify DaemonSet shows all nodes scheduled and ready: $ oc get ds cilium -n cilium # Expected: DESIRED == CURRENT == READY == UP-TO-DATE == AVAILABLE
GPU type confirmed and matching AI Defense connector selection
When deploying the connector, you will be prompted to select your GPU type. Confirm which GPU type your worker nodes use before starting the deployment.
criticalhardware

05

Required OCP Operators

5 items
NVIDIA GPU Operator installed and all GPU worker nodes show GPU resources
AI Defense Validation and Runtime processing requires GPU resources. The NVIDIA GPU Operator must be installed and all worker node GPUs must be schedulable before AI Defense is deployed.
criticalopenshifthardware
The NVIDIA GPU Operator installs drivers, the device plugin, and runtime components needed to expose GPUs as schedulable resources in Kubernetes. Without this, AI Defense pods that require GPU cannot be scheduled. Follow the NVIDIA installation guide for OpenShift: https://docs.nvidia.com/datacenter/cloud-native/openshift/latest/index.html
# Install via OperatorHub: # OCP Console → OperatorHub → search "NVIDIA GPU Operator" # Install into namespace: gpu-operator-resources # Verify after install: $ oc get pods -n gpu-operator-resources # All pods should be Running $ oc get nodes -L nvidia.com/gpu.present,nvidia.com/gpu.count # GPU presence and count visible on worker nodes $ oc get nodes -o json | jq '.items[].status.allocatable | with_entries(select(.key | contains("nvidia")))' # Each worker node should show nvidia.com/gpu: "4" or "8" # GPU smoke test — confirm a container can request and access a GPU: $ cat <<EOF | oc apply -f - apiVersion: v1 kind: Pod metadata: name: gpu-test spec: restartPolicy: Never containers: - name: cuda-test image: registry.access.redhat.com/ubi8/ubi:latest command: ["nvidia-smi"] resources: limits: nvidia.com/gpu: 1 EOF $ oc get pod gpu-test -w $ oc logs gpu-test # Expected: nvidia-smi table showing GPU model, driver version, and HBM memory $ oc delete pod gpu-test
Ingress NGINX Operator installed and an IngressClass created
AI Defense Runtime requires an Ingress controller to expose the AI Defense proxy gateway and API endpoint to on-prem AI applications. NGINX is the validated option in the CVD.
criticalopenshift
The AI Defense Runtime use case requires an Ingress controller to expose two services externally: the ai-defense-proxy-service (gateway mode) and ai-defense-apigateway (API inspection mode). These are accessed by AI applications on the on-prem network. Without a working Ingress controller, Runtime protection cannot be configured.
Ingress is only required for Runtime protection. If you plan to use only Validation (red teaming / security testing) in the first phase, this can be deferred. However install it before attempting any Runtime setup.
# Install NGINX Ingress via OperatorHub: # OCP Console → OperatorHub → search "NGINX Ingress Operator" # After install, create a NginxIngressController CR: $ cat <<EOF | oc apply -f - apiVersion: charts.nginx.org/v1alpha1 kind: NginxIngressController metadata: name: my-nginx-ingress-controller namespace: nginx-ingress spec: type: deployment nginxPlus: false ingressClass: nginx EOF # Verify: $ oc get pods -n nginx-ingress $ oc get ingressclass # Should show: nginx nginx.org/ingress-controller
If using H100 with time-slicing: devicePlugin config added to NVIDIA GPU Operator Helm values
H100 single-GPU time-slicing requires a devicePlugin configuration block in the NVIDIA GPU Operator Helm values before the connector is deployed. This must be done before the AI Defense install — it cannot be applied retroactively.
hardwareopenshifth100 only
Time-slicing allows multiple AI Defense workloads to share a single H100 GPU (up to 4 replicas). If you plan to use this option, the devicePlugin block must be present in the GPU Operator Helm values at install time. If you are using L40S GPUs or not using time-slicing, skip this item.
# Add to aid-nvidia-gpu-operator-values.yaml before helm install: devicePlugin: config: name: time-slicing-config create: true default: default data: default: |- version: v1 flags: migStrategy: none sharing: timeSlicing: renameByDefault: false resources: - name: nvidia.com/gpu replicas: 4
GPU worker nodes labeled with node_group=dps before Helm install
AI Defense uses the node_group=dps label as a required node selector to schedule its GPU workloads on the correct worker nodes. Without this label, AI Defense pods may attempt to schedule on control plane nodes (no GPUs) and fail.
criticalopenshifthardware
dps stands for Data Protection Service — the internal name for the AI Defense data plane. This label is a hard requirement in the AI Defense Helm chart. It must be applied before running the install script. It is not applied automatically and will not be re-applied if a worker node is rebuilt.
# Apply to all GPU worker nodes: $ oc label node <gpu-worker-1> node_group=dps --overwrite $ oc label node <gpu-worker-2> node_group=dps --overwrite # Verify: $ oc get node <gpu-worker-1> -o jsonpath='{.metadata.labels.node_group}{"\n"}' # Expected: dps
This label is not persistent across node rebuilds. If a GPU worker is rebuilt or a new worker added, re-apply node_group=dps immediately. Consider adding it to a MachineConfig to make it permanent.
Jump server / bastion host set up with oc, kubectl, and Helm 3
All install commands run from a jump server with network access to the OCP API (port 6443) and outbound internet (port 443). The OpenShift CLI (oc), Kubernetes CLI (kubectl), and Helm 3 must all be installed and verified before starting.
criticalopenshift
All AI Defense install commands run from the jump server. It must have network access to the OCP API (port 6443) and outbound internet on port 443. See Phase 0 step 0.3 of the install guide for full setup instructions.
# Install oc and kubectl (OCP 4.18+): $ curl -LO "https://mirror.openshift.com/pub/openshift-v4/clients/ocp/stable-4.18/openshift-client-linux.tar.gz" $ tar -xzf openshift-client-linux.tar.gz && sudo mv oc kubectl /usr/local/bin/ # Install Helm 3: $ curl https://raw.githubusercontent.com/helm/helm/main/scripts/get-helm-3 | bash # Verify cluster reachability — DNS, TCP, HTTPS: $ nslookup api.<cluster-name>.<domain> # Must resolve to an IP address $ nc -zv api.<cluster-name>.<domain> 6443 # Connection succeeded! $ curl -sk https://api.<cluster-name>.<domain>:6443 -o /dev/null -w "%{http_code}\n" # 403 = reachable (expected) # Log in and verify cluster-admin: $ oc login https://api.<cluster-name>.<domain>:6443 --username=kubeadmin $ oc cluster-info $ oc auth can-i '*' '*' --all-namespaces # yes $ oc get nodes # All nodes Ready $ helm version --short # v3.x.x

TLS Certificates & Ingress Configuration

3 items
TLS certificate available for the AI Defense Ingress subdomain
The NGINX Ingress configuration for AI Defense requires TLS. A certificate for the subdomain used by the AI Defense proxy and API endpoints must be available.
network
The CVD NGINX configuration example incorporates the certificate/key directly into the OpenShift Route. Customers may also use a domain-wide wildcard certificate. Either approach works but you must have a valid certificate before configuring the Ingress routes after connector deployment.
# Option A — self-signed (lab/test only): $ openssl req -x509 -nodes -days 365 -newkey rsa:2048 \ -keyout ai-defense.key -out ai-defense.crt \ -subj "/CN=<subdomain.domain.com>" # Option B — use existing wildcard cert # Option C — use cert-manager to issue from internal CA # Store cert and key — needed when creating the Ingress route: # cert → ai-defense.crt # key → ai-defense.key
Subdomain and DNS name chosen for AI Defense services
Both AI Defense services (proxy gateway and API endpoint) share a subdomain. Choose and document this before install — it is referenced in the Ingress config and must match the DNS entry added after deployment.
network
# Decide your subdomain, e.g.: # ai-defense.apps.<cluster-domain> # or a custom subdomain like: aidefense.<your-domain> # This subdomain will be used for two paths: # <subdomain>/api/v1/inspect → ai-defense-apigateway service # <subdomain>/ → ai-defense-proxy-service # After connector deployment, add DNS record: # <subdomain> → <IP of OCP Ingress Controller> # Get Ingress Controller IP after OCP install: $ oc get svc -n nginx-ingress # Note the EXTERNAL-IP of the nginx ingress service
AI application can be configured to point to AI Defense proxy URL
For Runtime protection (gateway mode), the AI application must be modified to route its LLM API calls through the AI Defense proxy instead of directly to the model. Confirm you have access to reconfigure the application.
network
The AI Defense gateway acts as a proxy between your AI application and the LLM. To use this, you redirect the application's LLM endpoint URL to the AI Defense proxy URL. If you cannot modify the application's configuration to change this URL, the gateway Runtime mode cannot be used — you would need to use the API inspection mode instead.
# In AI Defense UI (after connector is deployed): # Applications → Add Application → Connection Type: Gateway # The Gateway URL will be: <subdomain.domain.com>/<path> # Your AI application must be updated to use this URL # instead of calling the LLM directly

Final Pre-Install Gate

6 items
Beta limitation acknowledged: single hybrid connector per tenant
A hybrid tenant supports only one AI Defense hybrid connector. Plan your deployment accordingly — you cannot run multiple parallel connectors on the same tenant.
saasscc
AI Defense hybrid deployment is currently supported as a beta feature. A hybrid tenant is limited to hybrid operation only, and a SaaS tenant is limited to SaaS operation only — these cannot be mixed.
Beta limitation acknowledged: connector version must be n or n-1
AI Defense only supports the current connector version and the immediately preceding version (n and n-1). Ensure your connector will be kept up to date after deployment — outdated connectors are not supported.
saas
If you create a connector but do not complete the Helm install and leave it untouched, it may go out of date. In that case, delete the connector and begin a new deployment with the latest version.
All cluster operators Available, 0 Degraded, 0 Progressing
The OCP cluster must be fully stable before adding the AI Defense workload. Any degraded operator or progressing update must be resolved first.
criticalopenshift
$ oc get co | grep -v "True.*False.*False" # No output = all operators healthy. Ready to proceed. # Any output = do not proceed until resolved
Jump server can reach OCP API and has cluster-admin role
The AI Defense Helm install runs from the OCP jump server. The user account running the install must have cluster-admin privileges.
criticalopenshift
$ oc whoami $ oc auth can-i '*' '*' --all-namespaces # Must return: yes $ helm version $ helm list -A # No errors = ready for install
SCC steps from AI Defense UI obtained and ready to apply
Before running the Helm command, log into AI Defense → Administration → Hybrid Deployment → Deploy a Connector and copy the SCC configuration commands. These must be run first.
criticalscc
The CVD procedure states: run SCC commands FIRST, then the Helm install command. Reversing this order will cause the connector pods to fail with permission errors.
# In AI Defense SCC: # 1. Go to: AI Defense → Administration → Hybrid Deployment # 2. Click "Deploy a Connector" # 3. Follow prompts to see the SCC configuration commands # 4. Copy and save the SCC commands # 5. Copy and save the Helm install command (with API key embedded) # Run SCC commands first, then Helm command second
Connector status in AI Defense UI ready to show Active after install
The Deploy a Connector screen in AI Defense UI will show a pending connector. After running the Helm install and refreshing the page, the status should change to Active. Have this page open during install.
scc
# During install — have AI Defense UI open in browser # After helm install completes on the jump server: # Refresh AI Defense → Administration → Hybrid Deployment # Connector status: Pending → Active # If it stays Pending after 5 minutes: $ oc get pods -n ai-defense $ oc logs -n ai-defense <connector-pod> --tail=50 # Common causes: port 443 blocked, wrong API key, SCC not applied
