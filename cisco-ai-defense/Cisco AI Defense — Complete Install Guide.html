Pre-flight Verification
Confirm all prerequisites are in place before touching anything
0.1Verify OCP cluster health — all nodes Ready, all operators greenVerify

Do not proceed if any node is NotReady or any cluster operator is degraded. AI Defense adds workloads to all worker nodes — pre-existing issues will be compounded.

$ oc get nodes # All nodes must show STATUS=Ready $ oc get co | grep -v "True.*False.*False" # No output = all operators healthy. Ready to proceed. $ oc version # Server Version: 4.18.x or higher (hard requirement)
0.2Confirm outbound internet access on port 443 from OCP nodesVerify

The AI Defense connector communicates with the SaaS control plane via a bidirectional gRPC stream encrypted with TLS over port 443. This is the only outbound port required — but it must be fully unrestricted (no TLS inspection/MITM proxy).

$ curl -sk https://security.cisco.com -o /dev/null -w "SCC: %{http_code}\n" $ curl -sk https://us.aidefense.security.cisco.com -o /dev/null -w "AI Defense: %{http_code}\n" # Any HTTP response (200, 30x, 40x) = port 443 is open # curl: (000) or (35) = blocked at ACI or firewall — fix before proceeding
⚠
TLS inspection will break the gRPC stream
If your firewall performs SSL/TLS inspection on port 443, the gRPC connection between the on-prem connector and the AI Defense cloud gateway will fail. Add *.security.cisco.com and *.aidefense.security.cisco.com to your inspection bypass list.
0.3Confirm Helm 3 and oc CLI are installed on jump serverVerify
$ helm version --short # v3.x.x $ oc whoami # Must return your username $ oc auth can-i '*' '*' --all-namespaces # yes (cluster-admin required) # Install Helm if missing: $ curl https://raw.githubusercontent.com/helm/helm/main/scripts/get-helm-3 | bash
0.4Verify the cluster pull secret is valid and covers all required registriesVerify

The pull secret authenticates OpenShift to Red Hat's container registries. Without a valid pull secret covering the right registries, AI Defense operator images, the NVIDIA GPU Operator, and OCP operator updates will all fail with ImagePullBackOff. This is the most common silent pre-existing issue — verify it before proceeding.

# Step 1 — Check the pull secret has all required registries: $ oc get secret pull-secret -n openshift-config \ -o jsonpath='{.data.\.dockerconfigjson}' | base64 -d | \ python3 -m json.tool | grep -o '"[^"]*registry[^"]*"' # Must include all three: "registry.redhat.io" ← OCP core images, operators "quay.io" ← AI Defense, Isovalent, NVIDIA images "registry.connect.redhat.com" ← Certified operator images (AI Defense) # Step 2 — Definitive test: pull a Red Hat image from the cluster: $ oc run pull-test --image=registry.redhat.io/ubi9/ubi:latest \ --restart=Never --command -- echo "pull success" $ oc logs pull-test # pull success $ oc delete pod pull-test # Step 3 — Check for existing ImagePull failures across the cluster: $ oc get pods -A | grep -E "ImagePull|ErrImage" # Any output = pull secret issue — fix before continuing # Step 4 — Verify marketplace pods are healthy (depend on pull secret): $ oc get pods -n openshift-marketplace # All Running — ImagePullBackOff here means pull secret is broken # Step 5 — After AI Defense is deployed, confirm operator images pulled: $ oc get pods -n ai-defense \ -o jsonpath='{range .items[*]}{.metadata.name}: {.status.containerStatuses[0].image}{"\n"}{end}' # Running pods = images pulled successfully $ oc get events -n ai-defense | grep -i "pull\|image\|unauthorized\|403" # Any 401/403 events = pull secret missing entitlement for that registry
⚠
How to update a broken or missing pull secret
Download a fresh pull secret from console.redhat.com → OpenShift → Downloads → Pull Secret, then apply it:
$ oc set data secret/pull-secret -n openshift-config \ --from-file=.dockerconfigjson=pull-secret.json # Allow 5–10 minutes for MCO to propagate to all nodes $ oc get co machine-config # Wait for PROGRESSING=False before continuing

1.2Install the NVIDIA GPU OperatorOCP Console

AI Defense Validation and Runtime processing uses GPU resources on the AI POD worker nodes. The NVIDIA GPU Operator must be installed and all GPUs must be schedulable before the AI Defense Helm chart is deployed.

Install via OCP Console OperatorHub

  1. Open the OCP Web Console → Operators → OperatorHub
  2. Search for NVIDIA GPU Operator
  3. Click the operator, then click Install
  4. Set Installation Mode: All namespaces
  5. Set Installed Namespace: gpu-operator-resources (create new)
  6. Set Update Approval: Automatic
  7. Click Install and wait for status to show Succeeded
# Verify GPU Operator is running: $ oc get pods -n gpu-operator-resources # All pods should be Running # Verify GPUs are visible as schedulable resources on worker nodes: $ oc get nodes -o json | jq '.items[].status.allocatable | with_entries(select(.key | contains("nvidia")))' # Each C845A worker should show: "nvidia.com/gpu": "4" or "8" # Double-check on a specific node: $ oc describe node <worker-node-name> | grep -A3 "Allocatable:" # nvidia.com/gpu: 4 (small config) or 8 (medium/large)
🚫
Do not proceed if GPUs are not showing
If nvidia.com/gpu does not appear in node allocatable resources, AI Defense Validation pods will fail to schedule. Common cause: GPU Operator installed but Node Feature Discovery (NFD) hasn't labelled the nodes yet. Wait 5 minutes and re-check.

1.3Install the NGINX Ingress OperatorOCP Console

The NGINX Ingress Operator exposes the AI Defense proxy gateway and API endpoint to on-prem AI applications. Required for Runtime protection. If you plan to use only Validation initially, this can be deferred — but install it before any Runtime configuration.

⚠️
Important — Two NGINX controllers exist. Use the F5/NGINX Inc. version only.
In November 2025 the Kubernetes community announced the retirement of the community-maintained Ingress NGINX controller (kubernetes/ingress-nginx). Best-effort maintenance ended March 2026 — it no longer receives security patches, bug fixes, or new releases.

There are two completely separate projects:
✗ kubernetes/ingress-nginx — the retired community version. Do NOT install this.
✓ nginxinc/kubernetes-ingress — the F5/NGINX Inc. maintained version. This is what the Cisco CVD references and what you should install from OperatorHub.

To verify which one you have after install:
$ oc get pods -n nginx-ingress -o jsonpath='{.items[0].spec.containers[0].image}' # Correct: nginx/nginx-ingress:x.x.x ← F5 version, actively maintained # Wrong: registry.k8s.io/ingress-nginx/... ← retired community version
Longer term: The Kubernetes community recommends migrating to Gateway API as the strategic replacement for all Ingress controllers. Plan this for a future phase after the AI Defense deployment is stable.

Install via OCP Console OperatorHub

  1. Open OCP Web Console → Operators → OperatorHub
  2. Search for NGINX Ingress Operator
  3. Click Install → Set namespace to nginx-ingress → Click Install
  4. After install, navigate to Operators → Installed Operators → NGINX Ingress Operator
  5. Click Create NginxIngressController
# Create the NginxIngressController CR: $ cat <<EOF | oc apply -f - apiVersion: charts.nginx.org/v1alpha1 kind: NginxIngressController metadata: name: ai-defense-nginx namespace: nginx-ingress spec: type: deployment nginxPlus: false ingressClass: nginx serviceType: LoadBalancer EOF # Verify NGINX is running and has an external IP: $ oc get pods -n nginx-ingress # nginx-ingress-controller-xxx 1/1 Running $ oc get svc -n nginx-ingress # Note the EXTERNAL-IP — needed for DNS configuration later $ oc get ingressclass # nginx nginx.org/ingress-controller <age>
1.4Prepare TLS certificate for AI Defense Ingress endpointCommand

A TLS certificate is required for the AI Defense proxy and API endpoint. Choose your subdomain now — this value cannot be changed without redeploying the Ingress configuration. See the separate TLS Certificate Guide for all three cert options.

# Decide your AI Defense subdomain, e.g.: AIDEF_HOST="aidefense.apps.<cluster-name>.<domain>" # Generate self-signed cert (for CU / lab — replace with CA-signed for production): $ openssl req -x509 -nodes -days 365 \ -newkey rsa:2048 \ -keyout ai-defense.key \ -out ai-defense.crt \ -subj "/CN=${AIDEF_HOST}" \ -addext "subjectAltName=DNS:${AIDEF_HOST}" # Verify cert: $ openssl x509 -in ai-defense.crt -text -noout | grep -E "Subject:|DNS:|Not After"
1.5Configure OCP Security Context Constraints for AI Defense service accountsCommand

AI Defense requires three service accounts to run as any user ID. Grant anyuid SCC to each service account before deploying the connector.

# From the OCP jump server, run all three commands: $ oc adm policy add-scc-to-user anyuid -z pii-analyzer-serviceaccount $ oc adm policy add-scc-to-user anyuid -z rde-operator $ oc adm policy add-scc-to-user anyuid -z ai-defense-orchestrator # Verify each service account has the anyuid SCC assigned: $ oc get scc anyuid -o jsonpath='{.users}' | tr ',' '\n' | grep ai-defense # system:serviceaccount:ai-defense:pii-analyzer-serviceaccount # system:serviceaccount:ai-defense:rde-operator # system:serviceaccount:ai-defense:ai-defense-orchestrator
⚠
Run these before the Helm install
These SCC bindings must exist before Phase 3 Helm deployment. If the Helm install runs first, the AI Defense pods will start in a CreateContainerConfigError state due to SCC violations.

The CVD explicitly states this. Running Helm before the SCC configuration will result in connector pods failing with permission errors. The SCC commands create the namespace, service accounts, and security context constraints that the Helm chart expects to already exist.

3.2Run the Helm install command from the AI Defense UICommand — Run Second

The Helm command is generated by the AI Defense UI and includes your connector name, API key, and the correct Helm chart repository URL. Use the exact command from the UI — the pattern below shows the expected structure.

Post-Install Configuration
TLS secret, NGINX Ingress route, DNS entry — required before Runtime can be used
4.1Create the TLS Secret in the ai-defense namespaceCommand

The ai-defense namespace was created by the Helm install in Phase 3. Now create the TLS secret in that namespace. The NGINX Ingress route will reference this secret.

# Create TLS secret from the cert/key generated in Phase 1.3: $ oc create secret tls ai-defense-tls \ --cert=ai-defense.crt \ --key=ai-defense.key \ -n ai-defense # Verify: $ oc get secret ai-defense-tls -n ai-defense # TYPE: kubernetes.io/tls # Confirm the cert CN matches your intended subdomain: $ oc get secret ai-defense-tls -n ai-defense \ -o jsonpath='{.data.tls\.crt}' | base64 -d | \ openssl x509 -noout -subject -ext subjectAltName
4.2Deploy the NGINX Ingress route for AI Defense servicesConfig

This Ingress routes HTTPS traffic from AI applications to the two AI Defense services: the proxy gateway (root path) and the API inspection endpoint. Replace aidefense.yourdomain.com with your actual subdomain.

apiVersion: networking.k8s.io/v1 kind: Ingress metadata: name: ai-defense-ingress namespace: ai-defense annotations: nginx.org/proxy-connect-timeout: "30s" nginx.org/proxy-read-timeout: "300s" # Long timeout for AI inference nginx.org/proxy-send-timeout: "300s" nginx.ingress.kubernetes.io/ssl-redirect: "true" nginx.ingress.kubernetes.io/backend-protocol: "HTTPS" spec: ingressClassName: nginx tls: - hosts: - aidefense.yourdomain.com secretName: ai-defense-tls rules: - host: aidefense.yourdomain.com http: paths: - path: /api/v1/inspect pathType: Prefix backend: service: name: ai-defense-apigateway port: number: 443 - path: / pathType: Prefix backend: service: name: ai-defense-proxy-service port: number: 443
# Apply the Ingress configuration: $ oc apply -f ai-defense-ingress.yaml # Verify it was accepted: $ oc get ingress -n ai-defense $ oc describe ingress ai-defense-ingress -n ai-defense
📋
Alternative: Use OpenShift Routes instead of Kubernetes Ingress

6.3Get the Gateway URL and reconfigure your AI applicationSCC UI + App Config

The Gateway URL is the URL your AI application must use instead of calling the LLM directly. AI Defense sits between the application and the model, inspecting all prompts and responses.

In the Application view

  1. Click the three dots (⋮) next to the connection you just created
  2. Select Connection Guide
  3. Copy the Gateway URL shown
  4. Replace the host in the Gateway URL with your AI Defense proxy service hostname configured in Phase 4: aidefense.yourdomain.com
  5. Update your AI application's LLM endpoint URL to point to this new address
ℹ
How gateway mode works
Your AI application sends LLM requests to the AI Defense proxy URL instead of directly to the model. AI Defense inspects the prompt, enforces policy (block/allow/alert), then forwards allowed requests to the actual model. Responses from the model are also inspected before returning to the application.
6.4Create a Runtime Protection PolicySCC UI

In SCC → AI Defense → Policies

  1. Click Create Policy
  2. Give the policy a name, e.g. AI POD Model Protection Policy
  3. Configure the guardrail rules based on your Validation report findings from Phase 5. Recommended starting rules:
    • Prompt Injection → Block
    • Sensitive Data (PII) → Block
    • Jailbreak Attempts → Block
    • Off-topic Responses → Alert
    • Malicious URLs → Block
  4. Set the enforcement mode: Block for high confidence, Alert for monitoring
  5. Click Save Policy
6.5Apply the Policy to the Application ConnectionSCC UI

In SCC → AI Defense → Applications → [Your Application]

  1. Click the three dots next to the connection
  2. Select Assign Policy
  3. Select the policy created in Step 6.4
  4. Click Assign
