01

Environment Validation

5 items
All nodes Ready with no pre-existing conditions
Any degraded or NotReady node before migration becomes critical during CNI switchover.
critical
A pre-existing node issue that might be tolerable under OVN becomes a hard blocker during CNI migration. Every node must be Ready before starting.
$ oc get nodes # All nodes must show STATUS=Ready $ oc describe nodes | grep -A5 "Conditions:"
All cluster operators Available with 0 Degraded
Degraded operators before migration cannot be distinguished from migration-caused issues afterward.
critical
$ oc get co | grep -v "True.*False.*False" # Any output = problem to fix before migrating
OpenShift version is 4.14 or higher
The Cilium OLM operator and network operator migration API require OCP 4.14+.
openshift
$ oc version # Server Version must be 4.14.x or higher
VMware snapshots taken of all node VMs
Hypervisor-level snapshots are the only complete rollback option if migration fails after node reboots.
criticalvsphere
Once nodes reboot with Cilium as the CNI, rolling back requires either a VMware snapshot or a full cluster rebuild. Take snapshots of every VM before starting and verify they are usable.
# In vSphere for each node VM: # Right-click VM → Snapshots → Take Snapshot # Name: pre-cilium-migration-YYYYMMDD # ✓ Include VM memory (recommended) # ✓ Quiesce guest file system if VMware Tools installed
Network operator and network config backed up
Backup both objects before any changes so you can restore OVN if needed.
critical
$ oc get network.operator cluster -o yaml > ~/network-operator-backup.yaml $ oc get network.config cluster -o yaml > ~/network-config-backup.yaml
02

OperatorHub Catalog Verification

3 items
Use package name cilium-enterprise — NOT isovalent
The Cisco CVD incorrectly specifies name: isovalent. The actual package is cilium-enterprise.
criticalopenshift
Using the wrong package name causes the OLM Subscription to be created successfully but immediately enter ConstraintsNotSatisfiable. No operator pod will ever appear — the failure is completely silent unless you inspect the subscription status YAML.
$ oc get packagemanifest -n openshift-marketplace | grep -i -E "isovalent|cilium" # Expected: cilium-enterprise Certified Operators
Lesson learnedThe CVD specifies name: isovalent. This package does not exist. Always verify from the catalog before creating a Subscription.
Use a versioned channel (1.14) — there is no stable channel
The CVD specifies channel: stable. This channel does not exist. Available channels are 1.10 through 1.14.
criticalopenshift
$ oc get packagemanifest cilium-enterprise -n openshift-marketplace \ -o jsonpath='{.status.channels[*].name}' # Use the highest version shown, e.g.: channel: "1.14"
Lesson learnedUsing channel: stable causes the same silent ConstraintsNotSatisfiable failure as the wrong package name.
Certified operators catalog is healthy
If the catalog source is unhealthy, OLM cannot resolve any Subscription.
openshift
$ oc get catalogsource -n openshift-marketplace $ oc get pods -n openshift-marketplace | grep certified
03

OpenShift Security & Auth Preparation

3 items
Grant privileged SCC to Cilium service accounts after operator installs
OpenShift's default SCCs block all Cilium pods from scheduling. The CVD does not mention this. Must be applied before creating CiliumConfig.
criticalopenshift
Cilium requires hostNetwork, hostPath volumes, and kernel capabilities including NET_ADMIN and SYS_ADMIN — all denied by restricted-v2 SCC. Without the privileged SCC binding, the DaemonSet creates zero pods. The failure only appears in oc describe daemonset Events, not in any pod log.
# Run AFTER operator pod is Running and CSV is Succeeded: $ oc adm policy add-scc-to-user privileged -z cilium -n isovalent $ oc adm policy add-scc-to-user privileged -z cilium-operator -n isovalent # Verify: $ oc get rolebinding -n isovalent | grep privileged
Lesson learnedThe DaemonSet shows DESIRED=4, CURRENT=0 with no pod logs to read. Always check DaemonSet Events when pods fail to appear.
Backup admin account exists that does not depend on Keycloak or OIDC
Keycloak and other OIDC providers will be restarted during migration to get Cilium IPs. Without a backup admin you will be locked out.
criticalopenshift
Any pod surviving the CNI migration retains an OVN-era IP. Keycloak must be restarted to get a Cilium IP. If your only cluster-admin login routes through Keycloak, you will be locked out the moment Keycloak restarts. An htpasswd-based backup admin with cluster-admin role is essential.
# Create backup admin if not already done: $ htpasswd -c -B -b localauth.htpasswd backup-admin MySecurePassword123 $ oc create secret generic htpasswd-secret \ --from-file=htpasswd=localauth.htpasswd -n openshift-config $ oc adm policy add-cluster-role-to-user cluster-admin backup-admin # Test BEFORE starting migration: $ oc login -u backup-admin -p MySecurePassword123 https://api.<domain>:6443
Lesson learnedWe were locked out mid-migration when Keycloak restarted. We recovered using a backup-admin account set up during an earlier Trident install session. Without it, recovery would have required the kubeconfig file or a VMware snapshot revert.
Locate the install-time kubeconfig or kubeadmin credentials as last resort
Know where your kubeconfig or kubeadmin password file is before starting. Last resort if all other auth fails.
critical
# Typical locations: ~/install/auth/kubeconfig ~/install/auth/kubeadmin-password # Test kubeconfig works now: $ KUBECONFIG=~/install/auth/kubeconfig oc whoami
04

RHCOS Filesystem & CNI Path Checks

3 items
Confirm /opt/cni is a read-only symlink — override to /var/lib/cni/bin
On RHCOS, /opt/cni symlinks to the immutable OSTree overlay. Cilium's default CNI bin path will fail with a cryptic "file exists" error.
criticalrhcos
The Cilium init container tries to create /opt/cni/bin but on RHCOS this path symlinks into /usr/lib/opt/cni — on the read-only OSTree filesystem. The mkdir fails silently producing "file exists". The fix is to set cni.binPath in CiliumConfig using the cilium: subchart prefix.
# Verify on a node: $ oc debug node/<node> -- chroot /host ls -la /opt/cni # Expected on RHCOS: lrwxrwxrwx /opt/cni -> ../../usr/lib/opt/cni # Required CiliumConfig setting (cilium: subchart prefix is mandatory): spec: cilium: cni: binPath: "/var/lib/cni/bin" confPath: "/var/run/multus/cni/net.d"
Lesson learnedSetting cni.binPath at the top level of the CiliumConfig spec is silently ignored. It must be nested under cilium: because the CNI chart is a Helm subchart named cilium inside the parent cilium-enterprise chart.
Disable Hubble UI in CiliumConfig or provide a valid hostname
The Hubble UI Helm subchart requires a hostname. The CVD example omits this, causing every CiliumConfig apply to fail permanently.
critical
The Hubble UI chart validates that either frontend.server.host or ingress is configured. Without it, the chart errors at template render time and enters a permanent ReleaseFailed loop. To disable you must use the subchart key syntax — setting hubble.ui.enabled: false inside the hubble: block is insufficient.
# Option A — disable for migration (simplest, recommended): spec: hubble: enabled: false hubble-ui: # subchart key — required enabled: false hubble-enterprise: # subchart key — required enabled: false # Option B — enable with hostname: $ oc get ingress.config cluster -o jsonpath='{.spec.domain}' spec: hubble-ui: enabled: true frontend: server: host: "hubble-ui.apps.<your-domain>"
Lesson learnedhubble.ui.enabled: false does NOT disable the Hubble UI subchart. You must use hubble-ui.enabled: false as a top-level key. The CVD has no mention of this requirement.
Confirm /var/lib/cni/bin parent path is writable on all nodes
The writable CNI destination must be accessible before Cilium init containers run.
rhcos
$ oc debug node/<node> -- chroot /host ls -la /var/lib/cni/ # Should show: bin/ multus/ networks/ results/ # If bin/ is missing, Cilium will create it automatically
05

Network Operator Migration Sequence

4 items
Apply migration as three separate object changes — not a single patch
Patching only network.operator is silently reverted. All three objects must be updated together: annotation, network.config, and the migration field.
criticalopenshift
The network operator has a safety gate that rejects CNI type changes without the migration annotation, producing "Not applying unsafe configuration change: cannot change default network type when not doing migration." All three patches must be applied in sequence.
# Step 1 — annotation (unlocks safety gate) $ oc annotate network.operator cluster \ networkoperator.openshift.io/ovn-to-cilium-migration="" --overwrite # Step 2 — network.config (authoritative desired state) $ oc patch network.config cluster --type merge \ -p '{"spec":{"networkType":"Cilium"}}' # Step 3 — network.operator migration field $ oc patch network.operator cluster --type merge \ -p '{"spec":{"migration":{"networkType":"Cilium"}}}'
Lesson learnedWe discovered through trial and error that all three objects must be updated. The CVD implies a single patch is sufficient — it is not.
Expect OVN rollout to hang — be ready to manually force removal
The OVN DaemonSet will get stuck with ovnkube-controller=false on all nodes. This is intentional but requires manual unblocking.
criticalopenshift
During migration, ovnkube-controller intentionally marks itself not-ready to signal handoff to Cilium. The DaemonSet rollout controller waits for this to complete — creating a deadlock. After ~10 minutes of no progress, manually scale OVN down.
# Check per-container readiness: $ oc get pods -n openshift-ovn-kubernetes -l app=ovnkube-node \ -o jsonpath='{range .items[*]}{.metadata.name}: {range .status.containerStatuses[*]}{.name}={.ready} {end}{"\n"}{end}' # If ovnkube-controller=false on all nodes after 10+ minutes: $ oc scale deployment ovnkube-control-plane \ -n openshift-ovn-kubernetes --replicas=0 $ oc patch daemonset ovnkube-node -n openshift-ovn-kubernetes \ -p '{"spec":{"template":{"spec":{"nodeSelector":{"non-existing":"true"}}}}}'
Reconfigure Multus readiness indicator immediately after OVN removal
After OVN is removed, Multus will CrashLoopBackOff waiting for 10-ovn-kubernetes.conf which no longer exists.
criticalopenshift
Multus's daemon config hardcodes the OVN readiness indicator file. With OVN gone, Multus crashes on every restart. Without Multus, no new pods can get network interfaces. This must be fixed the moment it is detected.
# Detect — check Multus logs for: # "still waiting for readinessindicatorfile @ .../10-ovn-kubernetes.conf" $ oc logs -n openshift-multus -l app=multus --tail=5 # Fix — patch the ConfigMap and restart: $ oc patch configmap multus-daemon-config -n openshift-multus \ --type merge -p '{ "data":{"daemon-config.json":"{\"chrootDir\":\"/hostroot\",\"logToStderr\":true, \"cniConfigDir\":\"/run/multus/cni/net.d\",\"multusConfigFile\":\"auto\", \"readinessIndicatorFile\":\"/run/multus/cni/net.d/05-cilium.conflist\", \"namespaceIsolation\":false, \"globalNamespaces\":\"default,kube-system,openshift-multus\"}"}}' $ oc rollout restart daemonset/multus -n openshift-multus
Lesson learnedCilium always writes its config as 05-cilium.conflist. This patch can be prepared in advance and applied immediately when OVN pods are removed.
Restart all long-running pods to get Cilium IPs — DNS first
Any pod surviving the migration retains its OVN-era IP. Cilium cannot route to these addresses. DNS must be restarted first or all other pods will also fail.
critical
Cilium allocates from a different CIDR range than OVN. Pods that survive without restarting retain old IPs that Cilium has no routes to. DNS breaks first because dns-default pods are long-running. Restart DNS, verify it works, then identify and restart any other surviving pods.
# Step 1 — restart DNS (critical, do this first): $ oc rollout restart daemonset/dns-default -n openshift-dns # Step 2 — find any remaining OVN-era pods: $ oc get pods -A -o wide | grep -E "10\.128\.|10\.129\.|10\.130\.|10\.131\." # Step 3 — restart affected workloads # Step 4 — verify DNS is working: $ oc run test --image=busybox --rm -it --restart=Never \ -- nslookup kubernetes.default.svc.cluster.local # Expected: Server: 172.30.0.10 / Address: 172.30.0.1
Lesson learnedKeycloak (OIDC auth) was still on an OVN IP and became unreachable, locking us out. Restart all critical infrastructure pods before verifying application workloads.

Post-Migration Verification

4 items
All Cilium agent pods are 1/1 Running on every node
One Cilium agent pod must be Running on each node before the migration can be considered complete.
critical
$ oc get pods -n isovalent -l k8s-app=cilium -o wide # All must be 1/1 Running, one per node $ oc exec -n isovalent -l k8s-app=cilium -c cilium-agent \ -- cilium status 2>&1 | head -10 # Kubernetes: Ok # Cilium: Ok # Controller Status: 22/22 healthy
OVN-Kubernetes namespace is empty
No OVN pods should remain after migration is complete.
$ oc get pods -n openshift-ovn-kubernetes # Expected: No resources found in openshift-ovn-kubernetes namespace.
DNS resolution verified from a freshly created pod
End-to-end test confirming pod scheduling, Cilium IP assignment, and service routing all work.
critical
$ oc run test --image=busybox --rm -it --restart=Never \ -- nslookup kubernetes.default.svc.cluster.local # Expected: # Server: 172.30.0.10 # Name: kubernetes.default.svc.cluster.local # Address: 172.30.0.1
All cluster operators Available with 0 Degraded
The definitive sign the migration completed successfully.
critical
$ oc get co | grep -v "True.*False.*False" # Should return no output $ oc get co network # AVAILABLE=True PROGRESSING=False DEGRADED=False
