Phase 1 — Installing the Cilium Operator
Discovering the wrong package name and non-existent channel in the CVD
SetupCreate namespace and OperatorGroup

Cilium Enterprise is installed via OLM into its own namespace. OpenShift requires an OperatorGroup to scope which namespaces the operator watches. Both must exist before the Subscription can be created.

Command
$ cat <<EOF | oc apply -f - apiVersion: v1 kind: Namespace metadata: name: isovalent --- apiVersion: operators.coreos.com/v1 kind: OperatorGroup metadata: name: isovalent-operatorgroup namespace: isovalent spec: targetNamespaces: - isovalent EOF
namespace/isovalent created operatorgroup.operators.coreos.com/isovalent-operatorgroup created
ErrorCVD subscription fails — no pods appear after 3 minutes

The CVD specifies name: isovalent and channel: stable. We applied exactly as written. The Subscription resource was accepted by the API server (no YAML errors) but no operator pod ever appeared — a sign OLM resolved the subscription but couldn't find the package.

Command — as written in Cisco CVD
$ cat <<EOF | oc apply -f - apiVersion: operators.coreos.com/v1alpha1 kind: Subscription metadata: name: isovalent namespace: isovalent spec: channel: stable name: isovalent source: certified-operators sourceNamespace: openshift-marketplace EOF
subscription.operators.coreos.com/isovalent created
⚠
No pods after 3 minutes
OLM accepted the object syntax but couldn't resolve the package to an operator bundle. The subscription is stuck in a failed resolution state.
DiagnoseInspect subscription status.conditions to find the real error

When OLM cannot resolve a subscription it records the failure reason in status.conditions. We fetch the full subscription YAML to read that field and get the exact error message rather than guessing.

Command
$ oc get subscription isovalent -n isovalent -o yaml
status: conditions: - message: 'constraints not satisfiable: no operators found in package isovalent in the catalog referenced by subscription isovalent' reason: ConstraintsNotSatisfiable type: ResolutionFailed
What this told us: The package name isovalent does not exist anywhere in the certified-operators catalog. OLM searched the full index and found nothing. The CVD package name is simply wrong.
DiagnoseSearch the catalog for the real package name and channels

The PackageManifest objects represent the catalog index. Grepping them for Cilium/Isovalent terms will surface what the actual package is called. Then we query that specific package for its available channels.

Find the package
$ oc get packagemanifest -n openshift-marketplace | grep -i -E "isovalent|cilium"
cilium-enterprise Certified Operators 23d
Find available channels
$ oc get packagemanifest cilium-enterprise -n openshift-marketplace \ -o jsonpath='{.status.channels[*].name}'
1.10 1.11 1.12 1.13 1.14
What this told us: The package is cilium-enterprise not isovalent. There is no stable channel — only versioned channels 1.10 through 1.14. We use 1.14 as the latest, which matches the version referenced in the CVD body text even though the YAML example is wrong.
FixRecreate subscription with correct package name and channel

OLM does not self-heal a failed subscription. We delete the broken one and recreate with the correct values. A stuck subscription blocks any new subscription for the same operator in the same namespace.

Commands
$ oc delete subscription isovalent -n isovalent $ cat <<EOF | oc apply -f - apiVersion: operators.coreos.com/v1alpha1 kind: Subscription metadata: name: cilium-enterprise namespace: isovalent spec: channel: "1.14" name: cilium-enterprise source: certified-operators sourceNamespace: openshift-marketplace EOF
subscription.operators.coreos.com/isovalent deleted subscription.operators.coreos.com/cilium-enterprise created

NAME PHASE cilium-enterprise.v1.14.22-x6d86c88 Succeeded NAME READY STATUS cilium-ee-olm-844b88746d-ntvxr 1/1 Running

Phase 2 — Deploying CiliumConfig
Three separate blocking issues: Hubble UI validation, RHCOS read-only filesystem, and SCC permissions
ErrorHubble UI chart validation rejects every CiliumConfig apply

We applied the CiliumConfig from the CVD with Hubble enabled. The operator kept looping — no Cilium agent pods appeared. We needed to read the operator logs to find out why the Helm install kept failing.

Diagnosis — operator logs filtered to errors only
$ oc logs -n isovalent deployment/cilium-ee-olm --tail=10 | grep -v OverrideValues | grep -v DEBUG
ERROR helm.controller Release failed "error": "failed to install release: execution error at (cilium-enterprise/charts/hubble-ui/templates/configmap.yaml:58:17): `frontend.server.host` OR (`ingress.enabled` AND `ingress.host`) required"
What this told us: The Hubble UI Helm subchart has a required field guard at configmap.yaml line 58. It refuses to render without a hostname or ingress config. The CVD enables Hubble UI without providing these fields. The operator loops every ~30 seconds trying to install a chart that will always fail this validation check.
Attempted fix — get cluster ingress domain and add hostname
$ oc get ingress.config cluster -o jsonpath='{.spec.domain}' # → apps.oslab.oslab # Added frontend.server.host: "hubble-ui.apps.oslab.oslab" to CiliumConfig # STILL failed — the YAML nesting didn't match the Helm value path
Fix — disable Hubble UI using correct Helm subchart keys
$ cat <<EOF | oc apply -f - apiVersion: cilium.io/v1alpha1 kind: CiliumConfig metadata: name: cilium namespace: isovalent spec: debug: enabled: false kubeProxyReplacement: "strict" hubble: enabled: false hubble-ui: # subchart key — disables the whole subchart enabled: false hubble-enterprise: # subchart key — disables enterprise hubble enabled: false EOF
Why hubble-ui: as a top-level key? Cilium Enterprise is a Helm parent chart with named subcharts. In Helm values, subchart configuration lives under the subchart's chart name as a top-level key. hubble.ui.enabled: false sets a value inside the parent chart's hubble block. hubble-ui.enabled: false disables the entire subchart. These are different things.
ErrorSCC admission blocks all Cilium pods from scheduling

CiliumConfig showed UpgradeSuccessful but the DaemonSet showed DESIRED=4, CURRENT=0. No pods were scheduling. We need to find out why the pods are being rejected — DaemonSet events will show the scheduler/admission error.

Diagnosis — DaemonSet events
$ oc describe daemonset cilium -n isovalent | grep -A3 "Events"
Error creating: pods "cilium-" is forbidden: unable to validate against any security context constraint: provider restricted-v2: .spec.securityContext.hostNetwork: Invalid: Host network not allowed provider restricted-v2: .containers[0].capabilities.add: "NET_ADMIN" may not be added provider restricted-v2: .containers[0].capabilities.add: "SYS_ADMIN" may not be added provider "privileged": Forbidden: not usable by user or serviceaccount
What this told us: OpenShift's SCC admission evaluated the Cilium pod spec against every available SCC and rejected it from all of them. The Cilium agent needs hostNetwork, hostPath volumes, and privileged kernel capabilities — none permitted by default. Critically the privileged SCC shows "not usable by user or serviceaccount" — meaning the SCC exists but the Cilium service accounts haven't been bound to it. This is an RBAC binding gap, not a missing SCC.
Fix — bind the privileged SCC to Cilium service accounts
$ oc adm policy add-scc-to-user privileged -z cilium -n isovalent $ oc adm policy add-scc-to-user privileged -z cilium-operator -n isovalent
clusterrole.rbac.authorization.k8s.io/system:openshift:scc:privileged added: "cilium" clusterrole.rbac.authorization.k8s.io/system:openshift:scc:privileged added: "cilium-operator"
Why privileged SCC? Cilium is a CNI plugin operating at the kernel level via eBPF. It needs to mount BPF filesystems, access /proc, manage iptables, and use CAP_NET_ADMIN and CAP_SYS_ADMIN. These are the same capabilities OVN-Kubernetes already had. The CVD was written for upstream Kubernetes where SCCs don't exist.
ErrorRHCOS read-only filesystem blocks CNI binary installation

After SCC was granted, pods started but all hit Init:CreateContainerError. We need to identify which init container is failing and the exact error to understand the root cause.

Diagnosis — pod events
$ oc describe pod -n isovalent -l k8s-app=cilium | grep "Error\|Failed" | head -3
Error: failed to mkdir /opt/cni/bin: mkdir /opt/cni: file exists
Investigate the node filesystem structure
$ oc debug node/00-0c-29-e1-2f-26 -- chroot /host ls -la /opt/cni
lrwxrwxrwx. 1 root root 21 Mar 17 20:45 /opt/cni -> ../../usr/lib/opt/cni
Confirm the symlink target is on a read-only overlay
$ oc debug node/00-0c-29-e1-2f-26 -- chroot /host mkdir -p /usr/lib/opt/cni/bin
mkdir: cannot create directory '/usr/lib/opt/cni': Read-only file system
What this told us: On RHCOS, /opt/cni is a symlink to /usr/lib/opt/cni. The /usr tree is on an immutable OSTree overlay — read-only by design. The init container tries to mkdir /opt/cni which fails because the symlink already exists, and even if it didn't, the target is read-only. The writable path for CNI binaries on RHCOS is /var/lib/cni/bin which lives on the writable /var partition.
Fix — override CNI bin path in CiliumConfig with correct subchart prefix
$ cat <<EOF | oc apply -f - apiVersion: cilium.io/v1alpha1 kind: CiliumConfig metadata: name: cilium namespace: isovalent spec: debug: enabled: false kubeProxyReplacement: "strict" hubble: enabled: false hubble-ui: enabled: false hubble-enterprise: enabled: false cilium: # subchart prefix required for CNI values cni: binPath: "/var/lib/cni/bin" confPath: "/var/run/multus/cni/net.d" EOF
Why the cilium: prefix? The CiliumConfig maps to Helm values. Since cilium-enterprise wraps the main Cilium chart as a subchart named cilium, CNI values must be nested under cilium: to reach the subchart. Without the prefix, the values are ignored at the parent chart level.

Phase 3 — The CNI Cutover
Triggering the OVN → Cilium handoff and resolving the stuck rollout and Multus crash
DiagnoseConfirm current active CNI before patching

Despite Cilium agents running on all nodes, OVN is still authoritative. The network operator controls CNI selection. We verify the exact current state so we understand precisely what we are switching away from.

Command
$ oc get network.operator cluster -o jsonpath='{.spec.defaultNetwork.type}'
OVNKubernetes
FixApply the three-object migration sequence

We initially tried patching only network.operator — it was silently reverted. Then we tried patching only network.config — the operator rejected it as "unsafe configuration change: cannot change default network type when not doing migration." All three objects must be updated together to satisfy the operator's safety checks.

Step 1 — Migration annotation (unlocks the safety gate)
$ oc annotate network.operator cluster \ networkoperator.openshift.io/ovn-to-cilium-migration="" \ --overwrite
network.operator.openshift.io/cluster annotated
Step 2 — network.config (the authoritative desired-state object)
$ oc patch network.config cluster --type merge \ -p '{"spec":{"networkType":"Cilium"}}'
network.config.openshift.io/cluster patched
Step 3 — network.operator migration field (triggers migration controller)
$ oc patch network.operator cluster --type merge \ -p '{"spec":{"migration":{"networkType":"Cilium"}}}'
network.operator.openshift.io/cluster patched
Why three objects? network.config is the cluster's desired networking state. network.operator is the operator's own config. The migration field signals the operator to use a controlled migration procedure rather than an abrupt switch. The annotation unlocks a safety gate that prevents accidental CNI type changes. All three must align or the operator rejects the change.
ErrorOVN DaemonSet rollout hung — ovnkube-controller not ready on all 4 nodes

After the migration patch the network operator started progressing but then reported a hung rollout. We need to identify specifically which container is not ready and whether this is expected behavior or an actual error.

Diagnosis — per-container readiness on OVN pods
$ oc get pods -n openshift-ovn-kubernetes -l app=ovnkube-node \ -o jsonpath='{range .items[*]}{.metadata.name}: {range .status.containerStatuses[*]}{.name}={.ready} {end}{"\n"}{end}'
ovnkube-node-8pgdv: kube-rbac-proxy-node=true nbdb=true northd=true ovn-controller=true ovnkube-controller=false sbdb=true ovnkube-node-96kr2: kube-rbac-proxy-node=true nbdb=true northd=true ovn-controller=true ovnkube-controller=false sbdb=true ovnkube-node-pz4nt: kube-rbac-proxy-node=true nbdb=true northd=true ovn-controller=true ovnkube-controller=false sbdb=true ovnkube-node-tdmzm: kube-rbac-proxy-node=true nbdb=true northd=true ovn-controller=true ovnkube-controller=false sbdb=true
Diagnosis — check ovnkube-controller logs for actual errors
$ oc logs -n openshift-ovn-kubernetes ovnkube-node-8pgdv -c ovnkube-controller --tail=5
I0410 20:37:01 iptables.go:401] "Running" command="iptables-save" I0410 20:37:11 reflector.go:979] "Watch close" type="*v1.EgressIP" totalItems=10 I0410 20:38:01 reflector.go:979] "Watch close" type="*v1.NetworkPolicy" totalItems=10
What this told us: The ovnkube-controller is still running — the informer watch-closes are normal Kubernetes behavior, not errors. Its readiness probe is intentionally failing as the migration handoff mechanism — OVN signals to the network operator that it has yielded control. But the DaemonSet rollout controller was waiting for all pods to complete an update cycle that would never naturally finish. Manual intervention was required to unblock it.
Fix — scale down OVN control plane and evict node pods
$ oc scale deployment ovnkube-control-plane \ -n openshift-ovn-kubernetes --replicas=0 $ oc patch daemonset ovnkube-node \ -n openshift-ovn-kubernetes \ -p '{"spec":{"template":{"spec":{"nodeSelector":{"non-existing":"true"}}}}}'
deployment.apps/ovnkube-control-plane scaled daemonset.apps/ovnkube-node patched
ErrorMultus enters CrashLoopBackOff immediately after OVN removal

As soon as OVN pods were removed, all 4 Multus pods crashed. Multus is the primary CNI shim — without it, no new pods can get network interfaces. The cluster operator went Degraded. We need to diagnose fast.

Diagnosis — Multus logs
$ oc logs -n openshift-multus -l app=multus --tail=5
ERROR: still waiting for readinessindicatorfile @ /host/run/multus/cni/net.d/10-ovn-kubernetes.conf. pollimmediate error: timed out waiting for the condition
Verify what CNI configs exist on the node
$ oc debug node/00-0c-29-60-9f-c0 -- chroot /host ls -la /run/multus/cni/net.d/
-rw-r--r--. 1 root root 191 Apr 10 20:06 05-cilium.conflist -rw-------. 1 root root 252 Apr 10 20:39 10-ovn-kubernetes.conf.cilium_bak
What this told us: Cilium had already correctly written its CNI config (05-cilium.conflist) and renamed the OVN config with a .cilium_bak suffix. But Multus's daemon config hardcoded 10-ovn-kubernetes.conf as its readiness indicator — the specific file it waits for before marking itself Ready. With OVN gone, that file would never exist again. Multus would crash indefinitely on every restart cycle.
Fix — patch Multus daemon config to use Cilium readiness file
$ oc patch configmap multus-daemon-config -n openshift-multus \ --type merge -p '{ "data": { "daemon-config.json": "{\"chrootDir\":\"/hostroot\",\"logToStderr\":true, \"cniConfigDir\":\"/run/multus/cni/net.d\",\"multusConfigFile\":\"auto\", \"readinessIndicatorFile\":\"/run/multus/cni/net.d/05-cilium.conflist\", \"namespaceIsolation\":false, \"globalNamespaces\":\"default,kube-system,openshift-multus\"}" } }' $ oc rollout restart daemonset/multus -n openshift-multus
configmap/multus-daemon-config patched daemonset.apps/multus restarted
Path note: Multus uses chrootDir: /hostroot, meaning it prepends /hostroot to all config paths internally. The config must reference /run/multus/cni/net.d/05-cilium.conflist without a /host prefix — Multus resolves it against its chroot. The node directory listing confirmed this file exists at /run/multus/cni/net.d/ on the host.

ErrorDNS test times out — new pods can't reach cluster DNS

The definitive test of pod networking is creating a new pod and verifying it can resolve DNS. This exercises pod scheduling, CNI IP assignment, and service routing in a single command.

Test
$ oc run test --image=busybox --rm -it --restart=Never \ -- nslookup kubernetes.default.svc.cluster.local
;; connection timed out; no servers could be reached
Diagnosis — check what backend Cilium is routing DNS to
$ oc exec -n isovalent cilium-8q8vq -- cilium service list 2>&1 | grep "172.30.0.10"
62 172.30.0.10:53 ClusterIP 1 => 10.130.0.75:5353 (active)
Diagnosis — check actual DNS pod IPs
$ oc get pods -n openshift-dns -o wide
NAME IP NODE dns-default-pp62g 10.128.0.65 00-0c-29-60-9f-c0 (OVN-era IP) dns-default-t75b9 10.130.0.75 00-0c-29-00-5a-a2 (OVN-era IP)
Diagnosis — check Cilium's IPAM node CIDR allocations
$ oc exec -n isovalent cilium-8q8vq -- cilium node list 2>&1
Name IPv4 Address Endpoint CIDR 00-0c-29-00-5a-a2 10.7.1.82 10.0.2.0/24 00-0c-29-60-9f-c0 10.7.1.83 10.0.3.0/24 00-0c-29-6d-c4-20 10.7.1.84 10.0.1.0/24 00-0c-29-e1-2f-26 10.7.1.81 10.0.0.0/24
What this told us: Cilium allocated pod CIDRs in the 10.0.x.x/24 range — one /24 per node. DNS pods still had IPs from OVN's ranges (10.128-131.x.x). Cilium has no routes to these old addresses since they are outside its entire IPAM allocation. When a new pod (on a Cilium IP) queries 172.30.0.10:53, Cilium routes the request to 10.130.0.75 (the DNS backend) — but has no way to reach that address. The fix is to restart DNS pods so they receive new Cilium IPs, after which Kubernetes automatically updates the service endpoint and Cilium picks up the new backend.
FixRestart DNS pods — verify they receive Cilium IPs and service map updates
Restart DNS DaemonSet
$ oc rollout restart daemonset/dns-default -n openshift-dns
daemonset.apps/dns-default restarted
Verify new pods get Cilium IPs
$ oc get pods -n openshift-dns -o wide
NAME IP NODE dns-default-dqkt9 10.0.2.22 00-0c-29-00-5a-a2 dns-default-vtbpc 10.0.3.32 00-0c-29-60-9f-c0
Verify Cilium service map auto-updated
$ oc exec -n isovalent cilium-8q8vq -- cilium service list 2>&1 | grep "172.30.0.10"
62 172.30.0.10:53 ClusterIP 1 => 10.0.3.32:5353 (active)
What happened: When DNS pods restarted, Kubernetes updated the Endpoints/EndpointSlice objects for the dns-default service. Cilium's controller watches EndpointSlices and automatically updated its eBPF service map with the new backend IPs. No manual Cilium intervention was needed — this is Cilium's reconciliation loop working correctly.

Outstanding Items
Three items not yet completed — follow-up required
WarnKubeProxy Replacement in hybrid mode — eBPF service routing not active

Cilium status shows KubeProxyReplacement: False. Service routing is falling back to iptables instead of eBPF. Full replacement is one of Cilium's primary benefits.

Current state
$ oc exec -n isovalent cilium-8q8vq -- cilium status | grep KubeProxy
KubeProxyReplacement: False [br-ex (Direct Routing)]
ℹ
To enable full replacement
Update CiliumConfig: set kubeProxyReplacement: true, add k8sServiceHost: api.oslab.oslab and k8sServicePort: "6443". Also set deployKubeProxy: false on the network operator. Test during a maintenance window as this may briefly disrupt service routing.
WarnHubble UI disabled — needs proper hostname configuration to enable

Hubble UI was disabled to unblock the migration. It provides real-time L3–L7 network flow visibility and is one of the key value-adds of Cilium Enterprise.

ℹ
To enable Hubble UI
In CiliumConfig: set hubble-ui.enabled: true and hubble-ui.frontend.server.host: "hubble-ui.apps.oslab.oslab". Also set hubble.relay.enabled: true. Then create an OpenShift Route to expose the Hubble UI service externally.
WarnMultus ConfigMap change may not survive cluster upgrades

The multus-daemon-config ConfigMap is owned by the OpenShift network operator. During a cluster upgrade, the operator may reconcile this ConfigMap back to its original state, restoring the OVN readiness indicator path — which would crash Multus again since OVN is gone.

ℹ
Recommended action
Monitor this ConfigMap during the next cluster upgrade. If reverted, investigate whether an operator override annotation can prevent reconciliation. Consider opening a Red Hat support case — this is a gap in the official OVN-to-Cilium migration tooling for OpenShift.
