Certificate Architecture Overview

What Needs a Certificate

AI Defense Runtime requires two HTTPS endpoints exposed via NGINX Ingress on the OCP cluster. A TLS certificate is needed for the subdomain that fronts both services — either embedded directly in the Ingress route YAML, or via an OCP TLS Secret.

Why It Matters for AI Applications

AI applications on your on-prem network connect to AI Defense as a proxy. They make HTTPS calls to the AI Defense proxy URL. If the certificate is self-signed or untrusted, the AI application must be configured to trust it or connections will fail with TLS errors.

Certificate Trust Chain for AI Defense Ingress

Your internal PKI root CA — or a public CA (DigiCert, Let's Encrypt, etc.)

⚠

If your private key is password-protected, you must decrypt it before use in OpenShift. OCP does not support passphrase-protected keys in TLS Secrets. Run: openssl rsa -in encrypted.key -out decrypted.key

01

Choose Your Certificate Option Required Decision

The Cisco CVD notes that certificate incorporation "may vary depending on customer's environment." Three options are available. Choose the one that fits your environment — Option A is the fastest for a CU, Option B is preferred for production.

OptionMethodBest ForAI App TrustRenewal
A — Self-Signed Generated with openssl on jump server Lab, CU, proof of concept ✗ Must configure trust on AI apps Manual, annually
B — Internal CA Signed CSR submitted to internal PKI / AD CS Production, enterprise environments ✓ Trusted if CA in AD/OS trust store Manual or auto via ACME
C — Wildcard .apps Domain Use existing OCP wildcard cert When AI Defense subdomain is under .apps.* ✓ Same trust as rest of OCP apps Same cycle as OCP cluster certs
ℹ
Recommendation for CU
Use Option A (self-signed) for the CU to get the deployment running quickly, then replace with an internal CA-signed cert (Option B) before handing off to the customer for production use. Document the replacement procedure in your handoff notes.
02

Generate or Obtain the Certificate Required

2.1Decide on the subdomain / CN for the certificateInfo
Before generating anything, decide on the hostname. This same value is used in the cert CN/SAN, the NGINX Ingress route, and the DNS A record. Changing it later requires regenerating the cert and redeploying the Ingress route.
# Common patterns — pick one and stick with it: aidefense.apps.<cluster-name>.<domain> # under OCP apps wildcard aidefense.<your-internal-domain> # separate DNS entry ai-defense.<your-internal-domain> # hyphenated variant # This hostname must: # 1. Match the CN or SAN in your TLS certificate # 2. Be configured in your internal DNS pointing to OCP Ingress IP # 3. Be reachable from your AI applications on the on-prem network # Get OCP Ingress Controller IP (needed for DNS entry later): $ oc get svc -n nginx-ingress # Note EXTERNAL-IP — this is where the DNS A record points
2.2AOption A — Generate a self-signed certificateCommand
Generates a self-signed cert and key in PEM format valid for 1 year. Replace aidefense.yourdomain.com with your chosen subdomain.
# Generate private key and self-signed certificate in one step: $ openssl req -x509 -nodes -days 365 \ -newkey rsa:2048 \ -keyout ai-defense.key \ -out ai-defense.crt \ -subj "/C=US/ST=CA/O=YourOrg/CN=aidefense.yourdomain.com" \ -addext "subjectAltName=DNS:aidefense.yourdomain.com" # Verify the cert: $ openssl x509 -in ai-defense.crt -text -noout | grep -A2 "Subject\|SAN\|Validity" # Output files: ai-defense.crt ← certificate (PEM format) ai-defense.key ← private key (unencrypted PEM)
⚠
Self-signed trust issue
AI applications making HTTPS calls through the AI Defense proxy will receive a certificate from an untrusted CA. They must either be configured to skip TLS verification (insecure, not for production) or the self-signed cert's CA must be added to their trust store.
2.2BOption B — Generate a CSR and get it signed by internal CACommand
Generates a private key and a Certificate Signing Request (CSR) to submit to your internal PKI (AD CS, HashiCorp Vault, etc.).
# Step 1 — Generate private key: $ openssl genrsa -out ai-defense.key 2048 # Step 2 — Create CSR with SAN: $ cat > ai-defense-csr.conf <<EOF [req] default_bits = 2048 prompt = no default_md = sha256 distinguished_name = dn req_extensions = req_ext [dn] C = US ST = YourState O = YourOrganization CN = aidefense.yourdomain.com [req_ext] subjectAltName = @alt_names [alt_names] DNS.1 = aidefense.yourdomain.com EOF $ openssl req -new -key ai-defense.key -out ai-defense.csr -config ai-defense-csr.conf # Step 3 — Verify CSR contents: $ openssl req -text -noout -verify -in ai-defense.csr # Step 4 — Submit ai-defense.csr to your internal CA # Retrieve the signed certificate as ai-defense.crt (PEM format) # Also retrieve intermediate/root CA cert as ca-bundle.crt if needed # Step 5 — Verify signed cert: $ openssl verify -CAfile ca-bundle.crt ai-defense.crt # ai-defense.crt: OK
2.2COption C — Use existing OCP wildcard .apps certificateInfo
If your AI Defense subdomain is under *.apps.<cluster>.<domain> you can reuse the existing OCP cluster wildcard certificate. Extract it from the cluster.
# Check if a custom wildcard cert is configured: $ oc get ingresscontroller default -n openshift-ingress-operator \ -o jsonpath='{.spec.defaultCertificate}' # If custom cert is configured, extract it: $ oc get secret <secret-name> -n openshift-ingress \ -o jsonpath='{.data.tls\.crt}' | base64 -d > ai-defense.crt $ oc get secret <secret-name> -n openshift-ingress \ -o jsonpath='{.data.tls\.key}' | base64 -d > ai-defense.key # If using the default self-signed OCP wildcard: $ oc get secret router-certs-default -n openshift-ingress \ -o jsonpath='{.data.tls\.crt}' | base64 -d > ai-defense.crt $ oc get secret router-certs-default -n openshift-ingress \ -o jsonpath='{.data.tls\.key}' | base64 -d > ai-defense.key
ℹ
Only works if subdomain is under .apps.*
The wildcard cert covers *.apps.clustername.domain. If your AI Defense subdomain is aidefense.apps.clustername.domain this works. If it's a completely separate DNS name (e.g. aidefense.corp.com) you need Option A or B.
03

Create the OCP TLS Secret Required

3.1Create the ai-defense namespace if it doesn't existCommand
$ oc get namespace ai-defense 2>/dev/null || oc create namespace ai-defense # The Helm install creates this namespace — only needed if creating secret before Helm
3.2Create the TLS secret in the ai-defense namespaceCommand
The secret must be in the same namespace as the AI Defense services (ai-defense). The NGINX Ingress route YAML references this secret by name.
$ oc create secret tls ai-defense-tls \ --cert=ai-defense.crt \ --key=ai-defense.key \ -n ai-defense # Verify the secret was created correctly: $ oc get secret ai-defense-tls -n ai-defense $ oc describe secret ai-defense-tls -n ai-defense # Should show: Type: kubernetes.io/tls # Data: tls.crt and tls.key present # Verify cert content in secret: $ oc get secret ai-defense-tls -n ai-defense \ -o jsonpath='{.data.tls\.crt}' | base64 -d | \ openssl x509 -text -noout | grep -E "Subject:|DNS:|Not After"
⚠
Create this secret AFTER the Helm install
The Helm install creates the ai-defense namespace. If you create the secret before Helm runs it may be in the wrong namespace or get overwritten. Create the secret as the first post-install step, then apply the Ingress route YAML.
3.3If using a CA-signed cert — also add CA bundle to OCP cluster trustCommand
For AI applications to trust the certificate without manual configuration, the signing CA must be in the OCP cluster's trust bundle. This is required for Option B (internal CA).
# Add your internal CA to the OCP cluster trust bundle: $ oc create configmap internal-ca \ --from-file=ca-bundle.crt=./ca-bundle.crt \ -n openshift-config $ oc patch proxy/cluster \ --type=merge \ --patch='{"spec":{"trustedCA":{"name":"internal-ca"}}}' # This propagates the CA to all nodes via the MCO (takes 5-10 min) # Monitor propagation: $ oc get co machine-config # Wait for PROGRESSING=False
04

Configure NGINX Ingress Route for AI Defense Required for Runtime

4.1Understand the two AI Defense services to exposeInfo
Service NamePathPurposeUsed By
ai-defense-proxy-service/ (root)Gateway proxy — AI app sends LLM calls here instead of directly to modelAI applications (Runtime Gateway mode)
ai-defense-apigateway/api/v1/inspectAPI inspection endpoint — inspect prompts/responses programmaticallyAI applications (Runtime API mode)
ℹ
Both services share the same subdomain
Both services are exposed under a single subdomain with different paths. Your NGINX Ingress config routes to the appropriate backend service based on the path.
4.2Create the NGINX Ingress route YAML — cert embedded in routeConfig
This is the approach used in the Cisco CVD — the certificate and key are embedded directly into the Ingress route. Replace aidefense.yourdomain.com with your chosen subdomain, and replace the cert/key values with base64-encoded content.
# Get base64-encoded cert and key values for embedding: $ cat ai-defense.crt | base64 -w 0 $ cat ai-defense.key | base64 -w 0 # You'll insert these values into the YAML below
apiVersion: networking.k8s.io/v1 kind: Ingress metadata: name: ai-defense-ingress namespace: ai-defense annotations: nginx.org/proxy-connect-timeout: "30s" nginx.org/proxy-read-timeout: "300s" # important for long AI inference calls nginx.ingress.kubernetes.io/ssl-redirect: "true" spec: ingressClassName: nginx tls: - hosts: - aidefense.yourdomain.com secretName: ai-defense-tls # references the TLS secret from Step 3.2 rules: - host: aidefense.yourdomain.com http: paths: - path: /api/v1/inspect pathType: Prefix backend: service: name: ai-defense-apigateway port: number: 443 - path: / pathType: Prefix backend: service: name: ai-defense-proxy-service port: number: 443
# Apply the Ingress configuration: $ oc apply -f ai-defense-ingress.yaml # Verify it was created: $ oc get ingress -n ai-defense $ oc describe ingress ai-defense-ingress -n ai-defense
4.3Alternative — embed cert/key directly in route (CVD approach)Config
The CVD example embeds cert and key directly in the route object rather than referencing a Secret. Both approaches work — use whichever matches your team's preference. The Secret reference approach (Step 4.2) is more maintainable for cert rotation.
# Create OCP Route directly with embedded cert (CVD method): $ cat <<EOF | oc apply -f - apiVersion: route.openshift.io/v1 kind: Route metadata: name: ai-defense-proxy namespace: ai-defense spec: host: aidefense.yourdomain.com path: / to: kind: Service name: ai-defense-proxy-service port: targetPort: 443 tls: termination: edge certificate: | $(cat ai-defense.crt | sed 's/^/ /') key: | $(cat ai-defense.key | sed 's/^/ /') insecureEdgeTerminationPolicy: Redirect --- apiVersion: route.openshift.io/v1 kind: Route metadata: name: ai-defense-api namespace: ai-defense spec: host: aidefense.yourdomain.com path: /api/v1/inspect to: kind: Service name: ai-defense-apigateway port: targetPort: 443 tls: termination: edge certificate: | $(cat ai-defense.crt | sed 's/^/ /') key: | $(cat ai-defense.key | sed 's/^/ /') insecureEdgeTerminationPolicy: Redirect EOF
05

DNS Configuration Required

5.1Get the OCP Ingress Controller external IPCommand
# Get the external IP of the NGINX Ingress service: $ oc get svc -n nginx-ingress # Note the EXTERNAL-IP — this is the IP your DNS A record must point to # Alternative — if using OCP default router (HAProxy): $ oc get svc router-default -n openshift-ingress # Note the EXTERNAL-IP
5.2Add the DNS A record in your internal DNS (ACI environment)Config
Add the DNS record to your internal DNS server. In an ACI environment this is typically the same DNS server configured in your ACI EPG/BD for the OCP network.
# DNS record to add: aidefense.yourdomain.com IN A <NGINX Ingress EXTERNAL-IP> # Verify DNS resolution after adding the record (allow 60s for propagation): $ nslookup aidefense.yourdomain.com # Expected: Address: <your NGINX Ingress IP> $ dig aidefense.yourdomain.com +short # Expected: <your NGINX Ingress IP> # Test HTTPS reachability (this should work after Ingress route is applied): $ curl -sk https://aidefense.yourdomain.com/api/v1/inspect \ -o /dev/null -w "HTTP: %{http_code}\n" # Expected: HTTP 4xx (route exists, auth required) — NOT connection refused
06

Verify Certificate is Working Required

6.1Check certificate presented by the endpointVerify
# Check what cert is being served: $ openssl s_client -connect aidefense.yourdomain.com:443 \ -servername aidefense.yourdomain.com < /dev/null 2>/dev/null | \ openssl x509 -text -noout | grep -E "Subject:|Issuer:|Not After:|DNS:" # Expected output (self-signed example): Subject: CN=aidefense.yourdomain.com Issuer: CN=aidefense.yourdomain.com Not After : Apr 10 2027 ... DNS:aidefense.yourdomain.com # Test with curl — check cert validity: $ curl -v https://aidefense.yourdomain.com/api/v1/inspect 2>&1 | \ grep -E "SSL|certificate|expire|verify" # For self-signed (ignore cert errors to confirm routing works): $ curl -sk https://aidefense.yourdomain.com/ -w "\nHTTP: %{http_code}\n" # Should return HTTP: 200 or 401/403 (not 000 connection refused)
6.2Verify AI Defense services are reachable through the IngressVerify
# Check both AI Defense service endpoints are routable: $ curl -sk https://aidefense.yourdomain.com/ \ -w "\n--- Proxy service HTTP: %{http_code} ---\n" $ curl -sk https://aidefense.yourdomain.com/api/v1/inspect \ -w "\n--- API gateway HTTP: %{http_code} ---\n" # Check AI Defense pods are running and healthy: $ oc get pods -n ai-defense # All pods should be Running # Check services exist: $ oc get svc -n ai-defense | grep -E "proxy-service|apigateway" # Check Ingress/Routes are properly configured: $ oc get ingress -n ai-defense $ oc describe ingress ai-defense-ingress -n ai-defense | grep -A10 "Rules\|TLS"
6.3Verify AI Defense connector shows Active in SCC after cert configurationVerify
# In SCC → AI Defense → Administration → Hybrid Deployment # The connector status must show: Active ✓ # If it shows Pending or Error, check connector logs: $ oc get pods -n ai-defense | grep connector $ oc logs -n ai-defense <connector-pod-name> --tail=30
✓
Cert configuration complete when all of these are true
TLS secret exists in ai-defense namespace · Ingress route applied · DNS resolving to correct IP · curl returns non-zero HTTP code (not connection refused) · AI Defense connector shows Active in SCC
07

Certificate Renewal Procedure Operational

7.1Check certificate expiry dateCommand
# Check expiry of the cert currently in the OCP secret: $ oc get secret ai-defense-tls -n ai-defense \ -o jsonpath='{.data.tls\.crt}' | base64 -d | \ openssl x509 -noout -enddate # notAfter=Apr 10 00:00:00 2027 GMT # Set a reminder 30 days before expiry # Self-signed certs default to 1 year with the openssl command above
7.2Replace an expired or expiring certificateCommand
Generate the new cert (repeat Section 02 steps), then update the secret and trigger an Ingress controller refresh.
# Step 1 — Generate new cert (see Section 02 for your chosen option) # Step 2 — Update the TLS secret in-place: $ oc create secret tls ai-defense-tls \ --cert=ai-defense-new.crt \ --key=ai-defense-new.key \ -n ai-defense \ --dry-run=client -o yaml | oc apply -f - # Step 3 — Restart NGINX Ingress pods to pick up the new cert: $ oc rollout restart deployment -n nginx-ingress # Step 4 — Verify new cert is being served: $ openssl s_client -connect aidefense.yourdomain.com:443 \ -servername aidefense.yourdomain.com < /dev/null 2>/dev/null | \ openssl x509 -noout -enddate # Should show new expiry date

Troubleshooting Reference

T1curl: (35) SSL connect error / TLS handshake failedIssue
# Diagnose TLS handshake failures: $ openssl s_client -connect aidefense.yourdomain.com:443 \ -servername aidefense.yourdomain.com -debug 2>&1 | head -30 # Common causes: # 1. Private key in secret doesn't match the certificate: $ openssl x509 -noout -modulus -in ai-defense.crt | md5sum $ openssl rsa -noout -modulus -in ai-defense.key | md5sum # Both must return the SAME md5 hash # 2. Certificate CN/SAN doesn't match the hostname: $ openssl x509 -text -noout -in ai-defense.crt | grep -A1 "Subject Alt\|Subject:" # 3. Ingress controller not reloaded after secret update: $ oc rollout restart deployment -n nginx-ingress
T2curl: (000) connection refused — Ingress not routingIssue
# Check NGINX Ingress pods are Running: $ oc get pods -n nginx-ingress # Check Ingress resource was applied correctly: $ oc get ingress -n ai-defense $ oc describe ingress ai-defense-ingress -n ai-defense # Check AI Defense services exist and have endpoints: $ oc get svc -n ai-defense $ oc get endpoints -n ai-defense # Check DNS is resolving to the correct IP: $ dig aidefense.yourdomain.com +short # Must match NGINX Ingress EXTERNAL-IP, not the OCP HAProxy router IP
T3AI Defense connector stuck in Pending — not Active in SCCIssue
# Check connector pod logs: $ oc get pods -n ai-defense $ oc logs -n ai-defense <connector-pod> --tail=50 # Common causes: # 1. Outbound port 443 blocked to Cisco SCC (ACI contract or firewall) $ oc exec -n ai-defense <connector-pod> -- \ curl -sk https://security.cisco.com -o /dev/null -w "%{http_code}" # 2. Wrong API key used during Helm install # → Delete connector in SCC UI, generate new one, reinstall Helm # 3. SCC commands were not run before Helm install # → Check SCC for required commands, apply them, restart connector pods $ oc rollout restart deployment -n ai-defense
T4AI application gets certificate trust errors when calling AI Defense proxyIssue
# For self-signed cert (quick workaround — not for production): # Configure the AI application to skip TLS verification # Python requests example: # requests.post(url, verify=False) # For production — distribute the CA cert to the AI application hosts: # Copy ai-defense.crt (or ca-bundle.crt for Option B) to the AI app host # Linux — add to system trust: $ sudo cp ai-defense.crt /etc/pki/ca-trust/source/anchors/ $ sudo update-ca-trust # Python — use the cert file directly: # requests.post(url, verify='/path/to/ai-defense.crt') # If using Option B (internal CA), ensure the ROOT CA cert # (not the leaf cert) is distributed — the leaf cert alone is not sufficient
