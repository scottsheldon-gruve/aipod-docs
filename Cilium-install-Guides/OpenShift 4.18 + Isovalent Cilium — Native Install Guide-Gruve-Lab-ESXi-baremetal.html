How Cilium Works — eBPF, Identity & Labels
Read this before starting — understanding these concepts makes the install steps make sense
C.1The eBPF engine — networking at kernel speed, no iptablesConcept

Cilium is built entirely on eBPF (extended Berkeley Packet Filter) — a Linux kernel technology that runs sandboxed programs directly in the kernel without modifying kernel source code or loading kernel modules. Traditional CNIs like OVN-Kubernetes use iptables — a linear list that every packet must traverse, degrading at scale. Cilium's eBPF programs are compiled JIT and attached directly to kernel network hooks, executing at wire speed with zero userspace context switches.

ℹ
Why eBPF beats iptables
iptables is a linear list — every packet traverses every rule. At 10,000+ rules (common in large clusters) this causes measurable latency. eBPF uses hash maps — O(1) lookup regardless of rule count. Cilium benchmarks show 30%+ better throughput and lower latency vs. iptables-based CNIs at scale.
C.2Identity-based security — labels, not IP addressesConcept

Traditional firewalls use IP addresses and ports. Cilium uses identity — a numeric value derived from Kubernetes labels on pods, namespaces, and nodes. Policy is enforced against identities, not IPs — so policies survive pod restarts and IP changes automatically with zero rule updates.

How Identity is Assigned

When a pod starts, the Cilium agent reads its labels and computes a stable numeric identity, distributed to eBPF maps on every node cluster-wide.


app=nginxnamespace=prodtier=frontend

→ Cilium Identity: 12345 — programmed in eBPF on all nodes

How Policy Uses Labels

NetworkPolicy rules reference labels — not IPs. Cilium looks up the source identity in the eBPF map and checks against policy. No IP tracking required.


podSelector: matchLabels: app: backend # Matches any pod with app=backend # regardless of current IP address

Why This Matters

Pod IPs change constantly — restarts, scaling, rescheduling. IP-based rules need constant updates. Cilium label-based identity means:


✅ Policies survive pod restarts automatically

✅ New replicas get policy immediately

✅ No IP management overhead

✅ Works across nodes and clusters

Hubble Uses the Same Identity

Hubble reads the same eBPF hooks, so every captured flow has pod identity attached — pod name, namespace, and labels — not just raw IPs.


prod/nginx → prod/api:80 FORWARDED dev/client → prod/api:80 DROPPED # Policy denied — identity mismatch
C.3Cilium in OpenShift — Multus, OLM, and why day-0 mattersConcept

OpenShift mandates Multus as a meta-CNI on every cluster — it allows pods to have multiple network interfaces and delegates primary pod networking to whichever CNI is configured as the primary plugin. In this install, that is Cilium.

The Isovalent OLM operator manages the Cilium agent DaemonSet lifecycle. By placing the complete OLM manifests in the openshift/ directory before generating the agent ISO, the operator deploys during bootstrap — ensuring Cilium writes its CNI config to /run/multus/cni/net.d before Multus starts looking for it. This is what "native day-0 Cilium" means.

🚫
This is why the install order matters
If Multus starts before Cilium has written its CNI config, Multus enters CrashLoopBackOff. Every other pod in the cluster goes Pending. The only fix is to regenerate the ISO with the correct Isovalent manifests in openshift/ and reinstall. There is no in-place recovery.

1.3Install nmstatectl stub — Ubuntu workaroundWorkaround

The openshift-install binary requires nmstatectl for networkConfig validation. nmstatectl is not available in Ubuntu 24.04 apt repos. Since we use dnsmasq DHCP reservations for static IPs instead of networkConfig, a stub that satisfies the binary check is sufficient.

# Create nmstatectl stub $ sudo tee /usr/local/bin/nmstatectl << 'EOF' #!/bin/bash exit 0 EOF $ sudo chmod +x /usr/local/bin/nmstatectl $ hash -r $ nmstatectl --version # Silent exit = stub is working
ℹ
Why this works
The stub bypasses nmstate validation on the helper VM. The actual static IP assignment is handled by dnsmasq DHCP reservations — nodes boot, receive their reserved IPs by MAC address, and always land on the correct static IPs. No node-level networkConfig is needed.

2.2Create /etc/dnsmasq.d/cilium.conf with all required optionsConfig

This single file provides DNS resolution, DHCP IP assignment by MAC, default gateway, DNS server, and NTP server to all cluster nodes. Create it completely before booting any VMs.

$ sudo tee /etc/dnsmasq.d/cilium.conf << 'EOF' # DNS records for os-lab-cilium cluster address=/api.os-lab-cilium.oslab.oslab/10.7.1.160 address=/api-int.os-lab-cilium.oslab.oslab/10.7.1.160 address=/.apps.os-lab-cilium.oslab.oslab/10.7.1.159 address=/master1.os-lab-cilium.oslab.oslab/10.7.1.161 address=/master2.os-lab-cilium.oslab.oslab/10.7.1.162 address=/master3.os-lab-cilium.oslab.oslab/10.7.1.163 address=/worker1.os-lab-cilium.oslab.oslab/10.7.1.164 # DHCP range and MAC reservations dhcp-range=10.7.1.100,10.7.1.200,12h dhcp-host=00:0c:29:e1:2f:26,master1,10.7.1.161 dhcp-host=00:0c:29:00:5a:a2,master2,10.7.1.162 dhcp-host=00:0c:29:60:9f:c0,master3,10.7.1.163 dhcp-host=00:0c:29:6d:c4:20,worker1,10.7.1.164 # DHCP options: gateway, DNS, NTP dhcp-option=3,10.7.1.1 # Default gateway dhcp-option=6,10.7.1.75 # DNS server (this helper VM) dhcp-option=42,216.239.35.0 # NTP server (Google) # Upstream DNS forwarders server=8.8.8.8 server=1.1.1.1 EOF $ sudo systemctl restart dnsmasq
🚫
Include ALL dhcp-option lines before booting nodes
On the first install attempt, dhcp-option=3 (gateway) was missing. Nodes received IPs but had no default route, causing all quay.io pulls to time out with "dial tcp: i/o timeout." All three dhcp-option lines (gateway, DNS, NTP) must be present before the first node boot.

3.2Customize CiliumConfig — enable Hubble relay and UIConfig

The default CiliumConfig from the repo has Hubble defined but not fully enabled. Replace it with the version below which enables Hubble relay, UI, and kubeProxyReplacement. Also sets the OpenShift-specific VXLAN port (4789) and cluster health port (9940).

$ cat > ~/ocp-cilium-install/openshift/cluster-network-07-cilium-ciliumconfig.yaml << 'EOF' apiVersion: cilium.io/v1alpha1 kind: CiliumConfig metadata: name: cilium namespace: cilium spec: debug: enabled: false cni: binPath: "/var/lib/cni/bin" confPath: "/var/run/multus/cni/net.d" chainingMode: portmap securityContext: privileged: true kubeProxyReplacement: strict k8sServiceHost: api.os-lab-cilium.oslab.oslab k8sServicePort: 6443 clusterHealthPort: 9940 tunnelPort: 4789 etcd: leaseTTL: 30s hubble: enabled: true peerService: clusterDomain: cluster.local relay: enabled: true tls: auto: certValidityDuration: 1095 enabled: true method: helm ui: enabled: true ipam: mode: cluster-pool operator: clusterPoolIPv4MaskSize: "23" clusterPoolIPv4PodCIDRList: - "10.128.0.0/14" ipv4: enabled: true k8s: requireIPv4PodCIDR: true sessionAffinity: true prometheus: serviceMonitor: {enabled: false} EOF
ℹ
OpenShift-specific port requirements
tunnelPort: 4789 — OpenShift uses UDP 4789 (IANA standard) for VXLAN, not Cilium's default UDP 8472 (Linux). clusterHealthPort: 9940 — OpenShift blocks low-range ports so the health check must use a higher port.

What Hubble does
Hubble is Cilium's built-in observability layer. It uses the same eBPF hooks as Cilium to capture every network flow across the cluster in real time — with identity-aware context (pod name, namespace, labels) rather than just raw IPs. The relay aggregates flows from all nodes; the UI visualizes them in a service map.

8.4Use Hubble CLI for real-time flow observationCommand

The Hubble CLI runs inside the Cilium agent pods. Use it for real-time flow filtering, dropped packet analysis, and policy troubleshooting.

# Observe last 20 flows cluster-wide $ oc exec -n cilium ds/cilium -- hubble observe --last 20 # Follow flows in real time $ oc exec -n cilium ds/cilium -- hubble observe --follow # Filter to a specific namespace $ oc exec -n cilium ds/cilium -- hubble observe \ --namespace openshift-dns --follow # Show only dropped flows (policy violations) $ oc exec -n cilium ds/cilium -- hubble observe \ --verdict DROPPED --follow # Filter by destination port (e.g., all DNS traffic) $ oc exec -n cilium ds/cilium -- hubble observe \ --port 53 --follow # Filter between specific source and destination $ oc exec -n cilium ds/cilium -- hubble observe \ --from-namespace my-app --to-namespace openshift-dns --follow
8.5Port-forward Hubble relay for local Hubble CLI (alternative)Command

If you have the Hubble CLI installed locally on your Mac, you can port-forward the relay service and observe flows directly from your terminal without exec-ing into a pod.

# Install Hubble CLI on Mac $ brew install cilium/tap/hubble # Port-forward hubble-relay in one terminal $ oc port-forward -n cilium svc/hubble-relay 4245:4245 # In a second terminal — observe flows locally $ hubble observe --server localhost:4245 --follow $ hubble status --server localhost:4245 Healthcheck (via localhost:4245): Ok Connected Nodes: 4/4
ℹ
Hubble CLI vs Hubble UI — when to use which
Use the CLI for targeted troubleshooting — filter by namespace, port, verdict, or pod label in real time. Use the UI for a visual overview of service-to-service traffic patterns and to spot unexpected connections in your topology.

Troubleshooting
Known issues and resolutions from this install
T1Multus CrashLoopBackOff — "could not find a plugin configuration"Issue
failed to find the primary CNI plugin: could not find a plugin configuration in /host/run/multus/cni/net.d

Root Cause: Cilium manifests in openshift/ were incomplete. Cilium never fully initialized, so Multus had no CNI config file to read.

Fix: Always use the complete official manifests from isovalent/olm-for-cilium. Never hand-craft them. Re-run the install with the full manifest set.

T2quay.io pull timeout — "dial tcp: i/o timeout"Issue
pinging container registry quay.io: Get "https://quay.io/v2/": dial tcp 54.87.208.72:443: i/o timeout

Root Cause: DHCP option 3 (default gateway) was missing from dnsmasq. Nodes received IPs but had no route to the internet.

# Fix: add gateway option to dnsmasq and restart echo "dhcp-option=3,10.7.1.1" >> /etc/dnsmasq.d/cilium.conf sudo systemctl restart dnsmasq # Then reboot all nodes so they pick up the new DHCP option
T3NTP sync failures — "Host couldn't synchronize with any NTP server"Issue

Nodes reported NTP sync failures during bootstrap validation. Install proceeded but may cause certificate issues in production environments.

# Add NTP option to dnsmasq BEFORE booting nodes echo "dhcp-option=42,216.239.35.0" >> /etc/dnsmasq.d/cilium.conf sudo systemctl restart dnsmasq # Alternative: add to install-config.yaml before generating ISO additionalNTPSources: - time.cloudflare.com - 0.rhel.pool.ntp.org
T4nmstatectl not available on Ubuntu 24.04Issue

nmstatectl is not in Ubuntu apt repos and the pip install is broken on Ubuntu 24.04. GitHub release CDN downloads are also blocked in this lab network.

# Workaround: create a stub that exits 0 sudo tee /usr/local/bin/nmstatectl << 'EOF' #!/bin/bash exit 0 EOF sudo chmod +x /usr/local/bin/nmstatectl hash -r # Better long-term fix: use a RHEL/Rocky helper VM where nmstate installs cleanly sudo dnf install nmstate -y
T5install-config.yaml missing — "missing configuration or manifest file"Issue
FATAL failed to fetch Agent Installer ISO: failed to generate asset "Agent PullSecret": missing configuration or manifest file

Root Cause: The openshift-install binary consumed and deleted install-config.yaml and agent-config.yaml after the previous ISO generation run.

# Fix: recreate both files from your backups cp ~/install-config.yaml.bak ~/ocp-cilium-install/install-config.yaml cp ~/agent-config.yaml.bak ~/ocp-cilium-install/agent-config.yaml # Also clean up the previous install state cd ~/ocp-cilium-install rm -f agent.x86_64.iso rm -rf auth/ .openshift_install* .openshift_install_state.json
