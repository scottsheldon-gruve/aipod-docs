01 Executive Overview

This report documents the migration of an OpenShift 4.21.5 cluster running on VMware ESXi from OVN-Kubernetes to Isovalent Enterprise for Cilium (v1.14.22-cee.1), performed on April 10, 2026. The migration was guided by the Cisco FlashStack CVD (Cisco Validated Design) document for bare-metal OpenShift — however, the cluster environment is vSphere-based, which introduced several undocumented issues not covered in the CVD.

The migration was completed successfully after resolving eight distinct blocking issues. All cluster operators are healthy, all four nodes are Ready, OVN-Kubernetes has been fully removed, and pod networking and DNS resolution have been verified operational.

02 Issues Encountered & Resolutions

The following issues were encountered in sequence during the migration. Each required diagnosis and a manual fix not documented in the Cisco CVD.

01 Incorrect OperatorHub Package Name Blocker
Root Cause
The CVD instructs creating a Subscription with name: isovalent. This package does not exist in the certified-operators catalog. The subscription was created but immediately failed with "no operators found in package isovalent".
Fix Applied
Queried the packagemanifest catalog to find the correct name. Package is cilium-enterprise, not isovalent. Additionally the CVD specifies channel: stable — only versioned channels 1.10–1.14 exist. Used channel 1.14.
02 Hubble UI Missing Required hostname — CiliumConfig Rejected Blocker
Root Cause
The Isovalent Enterprise Helm chart for Hubble UI requires either frontend.server.host or ingress.enabled + ingress.host. The CVD's CiliumConfig example omits this, causing the Helm release to fail on every reconcile with an execution error at configmap.yaml:58.
Fix Applied
Disabled the Hubble UI subchart entirely for the migration using the correct subchart key syntax (hubble-ui.enabled: false, hubble-enterprise.enabled: false) to bypass chart validation. Hubble UI can be re-enabled post-migration with the correct hostname.
03 RHCOS Read-Only Filesystem — CNI Bin Path Blocked Blocker
Root Cause
On RHCOS nodes, /opt/cni is a symlink to /usr/lib/opt/cni which resides on a read-only filesystem overlay. The Cilium init container attempted mkdir /opt/cni and failed with "file exists" because the symlink itself exists but the bin directory inside it could not be created. The CVD makes no mention of this RHCOS filesystem constraint.
Fix Applied
Configured Cilium to use the writable CNI path /var/lib/cni/bin via the CiliumConfig spec using the correct subchart prefix: cilium.cni.binPath. This maps to the /var partition which is writable on immutable RHCOS nodes.
04 Missing SCC Privileges — Cilium DaemonSet Pods Blocked Blocker
Root Cause
OpenShift's Security Context Constraints blocked the Cilium DaemonSet from scheduling. Cilium requires host networking, hostPath volumes, and kernel capabilities (NET_ADMIN, SYS_ADMIN, SYS_MODULE, etc.) which are denied by the restricted-v2 SCC. The CVD does not address SCC configuration for OpenShift.
Fix Applied
Granted the privileged SCC to both the cilium and cilium-operator service accounts in the isovalent namespace using oc adm policy add-scc-to-user.
05 Network Operator Rejected Direct Type Change as Unsafe Config
Root Cause
Patching network.operator cluster with type: Cilium alone triggered the error: "Not applying unsafe configuration change: cannot change default network type when not doing migration." The migration annotation and the network.config object also needed to be updated together.
Fix Applied
Applied the migration annotation (networkoperator.openshift.io/ovn-to-cilium-migration: ""), patched network.config cluster with networkType: Cilium, and set spec.migration.networkType: Cilium on the network operator. All three were required together for the operator to accept the change.
06 OVN DaemonSet Rollout Hung — ovnkube-controller Not Ready Config
Root Cause
All four ovnkube-node pods showed 7/8 with the ovnkube-controller container intentionally marking itself not-ready as part of the handoff to Cilium. The network operator's rollout controller was blocking on this, waiting for the DaemonSet to complete an update that would never naturally complete.
Fix Applied
Manually scaled down the OVN control plane deployment to 0 replicas and removed OVN node pods from scheduling by patching the DaemonSet with a non-existent nodeSelector. This unblocked the network operator and allowed Cilium to take over fully.
07 Multus Waiting for OVN Readiness File After OVN Removal Blocker
Root Cause
After OVN was removed, Multus entered CrashLoopBackOff because its daemon config hardcoded readinessIndicatorFile: /host/run/multus/cni/net.d/10-ovn-kubernetes.conf. With OVN gone, this file no longer existed and Multus would never become ready.
Fix Applied
Patched the multus-daemon-config ConfigMap to update the readinessIndicatorFile to point to Cilium's config file: /run/multus/cni/net.d/05-cilium.conflist. Restarted the Multus DaemonSet to pick up the new config.
08 Existing Pods Retained OVN-Era IPs — DNS Broken After Migration Blocker
Root Cause
After the CNI switch, existing long-running pods (DNS, Keycloak, etc.) retained IP addresses from the old OVN CIDR ranges (10.128–131.x.x). Cilium's IPAM operates in the 10.0.x.x/24 per-node ranges and had no routes to the old OVN addresses. New pods got Cilium IPs but couldn't reach DNS pods on OVN-era IPs.
Fix Applied
Restarted all workload DaemonSets and Deployments with old IPs, starting with the critical dns-default DaemonSet. Once DNS pods received new Cilium IPs, the service endpoint updated automatically and DNS resolution was restored. All workloads were then verified on new 10.0.x.x Cilium IPs.

03 Gaps in the Cisco CVD

The Cisco FlashStack CVD was written for bare-metal RHCOS OpenShift. When applied to a VMware ESXi environment, the following critical topics were missing or incorrect.

Wrong Package Name

CVD uses name: isovalent in the Subscription. Correct name is cilium-enterprise. The stable channel also does not exist — only versioned channels 1.10–1.14.

Hubble UI Configuration Incomplete

CVD CiliumConfig example enables Hubble UI without the required frontend.server.host field, causing all Helm installs to fail. A working hostname or disabled UI is required.

No OpenShift SCC Guidance

CVD makes no mention of Security Context Constraints. On OpenShift (vs upstream Kubernetes), the privileged SCC must be explicitly granted to Cilium service accounts before any pods will schedule.

RHCOS Read-Only Filesystem Not Addressed

The CVD assumes /opt/cni/bin is writable. On RHCOS, this path is a symlink into a read-only overlay. The cni.binPath must be overridden to /var/lib/cni/bin.

Migration API Sequence Undocumented

The CVD implies a simple type patch on network.operator. In practice, three objects must be updated together: the migration annotation, network.config, and network.operator migration.networkType.

Multus Reconfiguration Not Mentioned

After OVN removal, Multus must be reconfigured to use Cilium's readiness indicator file. The CVD has no guidance on this. Without it, Multus crashes and all pod networking fails.

Post-Migration Pod Restart Required

The CVD does not mention that existing pods retain old CNI IPs after migration and must be restarted to receive Cilium IPs. This is critical for DNS and any long-running stateful workloads.

OVN Stuck Rollout — No Guidance

The CVD provides no guidance for the scenario where OVN's ovnkube-controller marks itself not-ready during migration and blocks the network operator rollout. Manual intervention is required to force OVN removal.

05 Red Hat Support Status

A critical question to understand before putting this migration into production context: is this process actually supported by Red Hat?

The Short Answer — It's a Grey Area

There are two distinct things to separate here: running Cilium on OpenShift, and the act of migrating to it on a live cluster. Red Hat treats these very differently.

Activity Red Hat Support Status
Cilium running on a self-managed OpenShift cluster ✓ Supported. Cilium holds Red Hat OpenShift CNI certification. Red Hat will support it and collaborate with Isovalent on issues.
Live migration from OVN-Kubernetes to Cilium ⚠ Not officially documented. Red Hat provides no migration guidance for switching to a third-party CNI on a running cluster. This is a grey area.
Migration from OpenShiftSDN to OVN-Kubernetes ✓ Fully supported. This is the only CNI migration Red Hat officially documents and supports.
Third-party CNI on managed OpenShift (ROSA, OSD, ARO) ✗ Not supported. Third-party CNIs including Cilium are not permitted on Red Hat managed services by default.

What This Means Practically

Once Cilium is running, Red Hat will support it — the certification covers Cilium operating within an OpenShift cluster. The unsupported part is the migration procedure itself. If a TAC case were raised mid-migration, Red Hat would likely decline to assist with the cutover steps. Large enterprise customers have performed this migration successfully by taking on the process themselves, which is exactly what was done here.

Since you are running Isovalent Enterprise (the Cisco/Red Hat certified version), you have dual support coverage once the migration is complete: Red Hat for OpenShift platform issues, and Isovalent for Cilium-specific issues. This is a better support position than running community Cilium.

Recommended Action

Notify your Red Hat account team that this migration has been performed so it is documented in your account record. This prevents ambiguity if you later raise a TAC case — support engineers will know Cilium is intentionally running as the CNI rather than flagging it as an anomaly.

06 Outstanding Items & Recommended Next Steps

Immediate

  • 01Enable full kube-proxy replacement. Update CiliumConfig with kubeProxyReplacement: true and set k8sServiceHost / k8sServicePort to the API server address. Current status is hybrid mode (eBPF + iptables fallback).
  • 02Enable Hubble UI with the correct hostname. Add hubble-ui.enabled: true and hubble-ui.frontend.server.host: hubble-ui.apps.oslab.oslab to the CiliumConfig. Create an OpenShift Route to expose it.
  • 03Persist the Multus configmap change. The multus-daemon-config patch may be overwritten by the network operator on cluster upgrades. Consider whether a MachineConfig or operator override is needed to make this durable.

Short Term

  • 04Validate all application workloads. Confirm that all namespaces have pods on Cilium IPs and that east-west traffic between services is functioning correctly. Pay particular attention to any services using NodePort or hostNetwork.
  • 05Test Cilium Network Policies. If any OVN-based NetworkPolicy resources were in place, verify they are being enforced by Cilium. Consider migrating to CiliumNetworkPolicy CRDs for richer L7 policy support.
  • 06Validate storage paths. Trident (NetApp) is running in the isovalent namespace — confirm PVC provisioning still works correctly post-migration and that storage traffic is unaffected.

Longer Term

  • 07Configure Hubble for observability. Once Hubble UI is enabled, set up flow visibility and consider integrating with Splunk or another observability platform for L3–L7 network telemetry.
  • 08Document the vSphere-specific deviation from the CVD. Create an internal runbook capturing all the fixes applied here so future cluster migrations or rebuilds don't hit the same issues.
  • 09Evaluate ClusterMesh if multi-cluster connectivity is needed. Cilium Enterprise supports ClusterMesh for cross-cluster service discovery — relevant if the ESXi environment hosts multiple OpenShift clusters.
