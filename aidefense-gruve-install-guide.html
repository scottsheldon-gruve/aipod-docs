0.3Validate LVM NVMe storage — cluster must be Healthy before proceedingVerify

The ai-defense-config-synchronizer-v2 (x3) pods require persistent volumes provisioned from the lvms-nvme storage class. If the LVM cluster is in a Failed state these pods will stay Pending. Verify storage is healthy before installing AI Defense.

# Check LVM cluster status: $ oc get lvmcluster -n openshift-storage # Status must show: Ready (not Failed) # Verify vg-manager DaemonSet is running on GPU workers: $ oc get daemonset -n openshift-storage # vg-manager DESIRED must NOT be 0 # vg-manager 0 0 0 0 0 = nodes missing lvms-nvme=true label → fix in Step 1 $ oc get pods -n openshift-storage -o wide # vg-manager pods should be Running on ocp-aipod1-wrk1 and ocp-aipod1-wrk2 # Check storage class is available: $ oc get storageclass lvms-nvme # Must exist and show (default) or be explicitly referenced

If vg-manager DESIRED=0 — apply node labels first
This was Issue 2 in the live deployment. If vg-manager shows DESIRED=0, the GPU worker nodes are missing the lvms-nvme=true label. Apply it in Step 1 before proceeding with the AI Defense install.

Installation Steps 1–6

Step 1Label GPU Worker Nodes — node_group=dps AND lvms-nvme=trueCommand

Two node labels are required on both GPU worker nodes. node_group=dps tells AI Defense which nodes to schedule its GPU workloads on (DPS = Data Protection Service). lvms-nvme=true enables the LVM vg-manager DaemonSet to manage NVMe volumes on those nodes for the config-synchronizer PVCs.

ℹ
Both labels must be applied — missing either one causes pod failures
Without node_group=dps → AI Defense GPU pods may schedule on control plane nodes (no GPU → fail). Without lvms-nvme=true → vg-manager DESIRED=0 → config-synchronizer PVCs never provision → pods Pending forever.
# Apply both labels to both GPU worker nodes: $ oc label node ocp-aipod1-wrk1.gruveai.com node_group=dps --overwrite $ oc label node ocp-aipod1-wrk2.gruveai.com node_group=dps --overwrite $ oc label node ocp-aipod1-wrk1.gruveai.com lvms-nvme=true --overwrite $ oc label node ocp-aipod1-wrk2.gruveai.com lvms-nvme=true --overwrite # Verify both labels are applied: $ oc get node ocp-aipod1-wrk1.gruveai.com -o jsonpath='{.metadata.labels.node_group}{"\n"}' # dps $ oc get node ocp-aipod1-wrk1.gruveai.com -o jsonpath='{.metadata.labels.lvms-nvme}{"\n"}' # true # Confirm vg-manager is now scheduling on workers: $ oc get daemonset vg-manager -n openshift-storage # DESIRED: 2 READY: 2 ← both GPU workers now managed
⚠
Labels are not persistent across node rebuilds
If either GPU worker node is rebuilt or a new worker node is added, both node_group=dps and lvms-nvme=true must be re-applied manually. Consider adding these labels to a MachineConfig to make them permanent.
Step 2Configure Security Context Constraints — grant anyuid to AI Defense service accountsCommand — Run Before Script

OpenShift's default restricted-v2 SCC blocks containers from running as arbitrary user IDs. AI Defense containers (especially the Triton GPU model servers) need to run with specific UIDs and create symbolic links in model directories. Grant anyuid SCC to the required service accounts.

🚫
Run these commands BEFORE the installation script in Step 5
If the script runs before SCCs are configured, Triton pods will start with restricted-v2 SCC and immediately crash with Permission denied on symlink creation. This was Issue 1 in the live deployment.
# Grant anyuid SCC to AI Defense service accounts in ai-defense-onprem namespace: $ oc adm policy add-scc-to-user anyuid \ -z default -n ai-defense-onprem $ oc adm policy add-scc-to-user anyuid \ -z ai-defense -n ai-defense-onprem $ oc adm policy add-scc-to-user anyuid \ -z ai-defense-operator -n ai-defense-onprem # Verify SCCs are applied: $ oc get rolebindings -n ai-defense-onprem | grep scc # Or check directly: $ oc adm policy who-can use scc/anyuid | grep ai-defense
ℹ
anyuid is scoped to the namespace
anyuid allows containers to run as any user ID but is scoped only to the specific service accounts in the ai-defense-onprem namespace — it is not granted cluster-wide.

Step 6Apply OpenShift Routes for aid-rte-onprem.gruveai.comConfig

Unlike the generic CVD guide which uses NGINX Ingress, the Gruve deployment uses native OpenShift Routes to expose the AI Defense services. Two routes are needed — one for the API gateway and one for the proxy service. TLS is terminated at the route using edge termination.

ℹ
OpenShift Routes vs NGINX Ingress
On Cisco AI PODs with ACI and Cilium CNI, OpenShift's built-in HAProxy router (Routes) is the simpler and more reliable option than adding an NGINX Ingress operator. Routes are native to OCP and don't require an additional operator.
# Route 1 — AI Defense API Gateway apiVersion: route.openshift.io/v1 kind: Route metadata: name: ai-defense-api namespace: ai-defense-onprem spec: host: aid-rte-onprem.gruveai.com path: /api/v1/inspect to: kind: Service name: ai-defense-apigateway port: targetPort: 443 tls: termination: edge insecureEdgeTerminationPolicy: Redirect --- # Route 2 — AI Defense Proxy Service apiVersion: route.openshift.io/v1 kind: Route metadata: name: ai-defense-proxy namespace: ai-defense-onprem spec: host: aid-rte-onprem.gruveai.com path: / to: kind: Service name: ai-defense-proxy port: targetPort: 443 tls: termination: edge insecureEdgeTerminationPolicy: Redirect
# Apply the routes: $ oc apply -f ai-defense-routes.yaml # Verify routes were created: $ oc get routes -n ai-defense-onprem NAME HOST PATH TERMINATION ai-defense-api aid-rte-onprem.gruveai.com /api/v1/inspect edge ai-defense-proxy aid-rte-onprem.gruveai.com / edge # Add DNS A record on 10.6.1.11: aid-rte-onprem.gruveai.com A 10.6.41.17 # Verify DNS resolves correctly: $ dig aid-rte-onprem.gruveai.com +short # 10.6.41.17 ← must match OCP Ingress VIP # Test HTTPS endpoints: $ curl -k https://aid-rte-onprem.gruveai.com/api/v1/inspect/health $ curl -k https://aid-rte-onprem.gruveai.com/

Real-World Issues Encountered & Resolved
4 issues hit during the live Gruve AI POD-1 deployment — documented with root cause and exact fix
Issue 1Triton GPU Pods CrashLoopBackOff — Permission Denied on symlink creationReal Issue

Symptom: After install, the three Triton model server pods (modernbert-pi-tox-triton-trt, modernbert-privacy-triton-trt, pi-mmbert-v2-triton-trt) immediately crash on startup.

# Symptom — pod logs show: ln: failed to create symbolic link '/models/classification/1/model.plan': Permission denied # Confirm SCC is the problem: $ oc get pod modernbert-pi-tox-triton-trt-xxx -n ai-defense-onprem \ -o jsonpath='{.metadata.annotations.openshift\.io/scc}{"\n"}' # restricted-v2 ← too restrictive for Triton containers

Root Cause: The Triton model server containers attempt to create a symbolic link at /models/classification/1/model.plan during startup. OpenShift's default restricted-v2 SCC blocked this because the containers needed to run with elevated permissions. The default service account in ai-defense-onprem had not been granted anyuid.

# Fix — grant anyuid to the default service account: $ oc adm policy add-scc-to-user anyuid \ -z default -n ai-defense-onprem # Delete the crashing pods to force restart with new permissions: $ oc delete pod modernbert-pi-tox-triton-trt-<pod-id> -n ai-defense-onprem $ oc delete pod modernbert-privacy-triton-trt-<pod-id> -n ai-defense-onprem $ oc delete pod pi-mmbert-v2-triton-trt-<pod-id> -n ai-defense-onprem # Verify pods come up Running: $ oc get pods -n ai-defense-onprem | grep triton # All three should show Running within 2-3 minutes # Confirm new pods are using anyuid: $ oc get pod modernbert-pi-tox-triton-trt-xxx -n ai-defense-onprem \ -o jsonpath='{.metadata.annotations.openshift\.io/scc}{"\n"}' # anyuid
Issue 2Config Synchronizer Pods Pending — LVM storage NotEnoughCapacityReal Issue

Symptom: All three ai-defense-config-synchronizer-v2 pods are stuck in Pending state after install.

# Symptom — pods in Pending: ai-defense-config-synchronizer-v2-0 0/1 Pending 0 5m ai-defense-config-synchronizer-v2-1 0/1 Pending 0 5m ai-defense-config-synchronizer-v2-2 0/1 Pending 0 5m # PVC error: $ oc describe pvc -n ai-defense-onprem | grep -A3 "Warning" Warning NotEnoughCapacity Requested storage (1Gi) is greater than available capacity on any node (). # Diagnose — check LVM cluster status: $ oc get lvmcluster -n openshift-storage # lvm-nvme Failed ← storage cluster is failed # Check why — vg-manager DaemonSet has DESIRED=0: $ oc get daemonset -n openshift-storage # vg-manager 0 0 0 0 0 lvms-nvme=true 72d ← no nodes have the label

Root Cause: The vg-manager DaemonSet has a node selector of lvms-nvme=true, but the GPU worker nodes never received this label when they joined the cluster. With DESIRED=0, no NVMe volume groups were managed and no PVCs could be provisioned.

# Fix — label both GPU worker nodes: $ oc label node ocp-aipod1-wrk1.gruveai.com lvms-nvme=true $ oc label node ocp-aipod1-wrk2.gruveai.com lvms-nvme=true # Verify vg-manager now schedules on workers: $ oc get daemonset -n openshift-storage # vg-manager 2 2 2 2 2 lvms-nvme=true ← both workers now managed # Watch vg-manager pods come up: $ oc get pods -n openshift-storage -o wide -w # vg-manager-xxx Running on ocp-aipod1-wrk1 # vg-manager-xxx Running on ocp-aipod1-wrk2 # Wait ~2 minutes for volume groups to initialize # Then verify LVM cluster is now Ready: $ oc get lvmcluster -n openshift-storage # lvm-nvme Ready ← storage is now healthy # Config synchronizer PVCs should now provision: $ oc get pvc -n ai-defense-onprem # All three PVCs: Bound # And the pods should start: $ oc get pods -n ai-defense-onprem | grep synchronizer # ai-defense-config-synchronizer-v2-0 1/1 Running
⚠
This label must be re-applied if nodes are rebuilt or new workers added
Consider adding lvms-nvme=true to a MachineConfig for worker nodes to make it permanent. Without it, any new or rebuilt worker will have DESIRED=0 on vg-manager again.
Issue 3triton-gemma-3-4b-it CrashLoopBackOff — TORCH_CUDA_ARCH_LIST missing for H100 HopperReal Issue

Symptom: The triton-gemma-3-4b-it pod CrashLoopBackOffs after the other Triton pods are fixed. The other three Triton pods run fine — this one is specific to the Gemma model.

# Symptom: triton-gemma-3-4b-it 0/1 CrashLoopBackOff # Error in pod logs: $ oc logs triton-gemma-3-4b-it-xxx -n ai-defense-onprem | tail -10 error: creating server: Internal - failed to load all models

Root Cause: The Gemma 3 4B Triton container requires TORCH_CUDA_ARCH_LIST=9.0 to be set for the NVIDIA H100 NVL GPU (Hopper architecture, compute capability 9.0). Without this environment variable, PyTorch cannot correctly target the GPU architecture during model compilation and the model fails to load.

# Fix — set the environment variable on the deployment: $ oc set env deploy/triton-gemma-3-4b-it \ -n ai-defense-onprem \ TORCH_CUDA_ARCH_LIST=9.0 # Trigger rollout to apply the change: $ oc rollout restart deploy/triton-gemma-3-4b-it -n ai-defense-onprem # Watch pod restart: $ oc rollout status deploy/triton-gemma-3-4b-it -n ai-defense-onprem -w # deployment "triton-gemma-3-4b-it" successfully rolled out # Verify env var is set: $ oc get deploy triton-gemma-3-4b-it -n ai-defense-onprem \ -o jsonpath='{.spec.template.spec.containers[0].env}' | jq . # [{"name":"TORCH_CUDA_ARCH_LIST","value":"9.0"}]
⚠
This env var may be overwritten by a Helm upgrade
After any AI Defense upgrade via the installation script, verify TORCH_CUDA_ARCH_LIST=9.0 is still present on the triton-gemma-3-4b-it deployment. If it's missing, re-apply the oc set env command above. This only affects H100 (Hopper) GPU deployments.
Issue 4HTTPS returns "Application is not available" — DNS pointing to wrong IPReal Issue

Symptom: All pods are Running, routes exist, but curl https://aid-rte-onprem.gruveai.com returns "Application is not available" from the OCP router.

# Diagnose — check what IP DNS is resolving to: $ dig aid-rte-onprem.gruveai.com # aid-rte-onprem.gruveai.com. 3600 IN A 10.6.41.37 ← WRONG # Should be: 10.6.41.17 (OCP Ingress VIP) # Confirm correct Ingress VIP: $ oc get ingresscontroller default -n openshift-ingress-operator \ -o jsonpath='{.status.endpointPublishingStrategy}' # Or check the router service: $ oc get svc router-default -n openshift-ingress # EXTERNAL-IP: 10.6.41.17 ← this is the correct target

Root Cause: The DNS A record for aid-rte-onprem.gruveai.com on DNS server 10.6.1.11 was pointing to 10.6.41.37 (incorrect) instead of 10.6.41.17 (OCP Ingress VIP).

# Fix — update the DNS A record on 10.6.1.11: aid-rte-onprem.gruveai.com A 10.6.41.17 # Verify after propagation (~60 seconds): $ dig aid-rte-onprem.gruveai.com +short # 10.6.41.17 ← now correct # Test HTTPS — should no longer return "Application is not available": $ curl -k https://aid-rte-onprem.gruveai.com/api/v1/inspect/health $ curl -k https://aid-rte-onprem.gruveai.com/ # Windows hosts file workaround (if DNS update is delayed): # Add to C:\Windows\System32\drivers\etc\hosts: 10.6.41.17 aid-rte-onprem.gruveai.com
